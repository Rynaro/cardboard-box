> SPECTRA planning artifact. Decision-ready. Confidence 0.87.

## 2. SCOPE

### 2.1 Complexity score (4-dimension matrix → 9/12)
| Dimension | Score | Rationale |
|---|---|---|
| Structural | 3/3 | New `tui/` subsystem: event loop, worker thread + channel, ≥6 screens, Model/Message/update/view. |
| Integration | 2/3 | Reuses existing `core::` + seam (no new core), but adds a concurrency boundary (worker thread) and a terminal-lifecycle boundary (alt-screen suspend/restore for `enter` + `$EDITOR`). |
| Uncertainty | 2/3 | Architecture (TEA) and deps are settled; the live risks are terminal restore on panic/signal and testing the TTY handoff without a PTY. |
| Stakes | 2/3 | A panic that leaves the terminal in raw mode is a bad UX, but `cbox` itself is recoverable; no data-loss surface beyond `rm` (which is already gated). |

→ 9/12, extended-thinking tier. **G3 (evaluator-optimizer)** considered; resolved single-pass at confidence ≥0.85 (see §11).

### 2.2 In scope (Phase 3 — deep)
- Entry: `cbox tui` (explicit) and `cbox` (no args) → launch TUI; feature-gate story; behavior when built without `tui`.
- TEA architecture: `Model`, `Message`, `update`, `view`, all in `src/tui/`; pure reducer unit-tested with no terminal.
- Event loop: crossterm input → `Message`; synchronous-by-default runner calls with a **background worker thread + channel** for long-running data actions (create/rm/apply/up/list-refresh) so the render never blocks.
- Screens: **Box list** (home), **Inspect/detail**, **Create wizard**, **Confirm-destroy modal**, **Apply/Up progress**, **Boxfile editor** (hand off to `$EDITOR` via alt-screen suspend — recommended).
- The **`enter` handoff**: alt-screen suspend → `core::enter` (`run_interactive`) → restore.
- Error/empty/loading states surfaced cozily, reusing `CboxError`; backend-unreachable leans on `core::doctor`.
- Acceptance criteria as GIVEN/WHEN/THEN against the pure reducer + `MockRunner`, mapped to `tests/` files.

### 2.3 Out of scope / deferred (v3.0)
- Mouse support, themes/colorschemes config, multi-pane tiling, async/streamed child-process *output* into the TUI (apply progress is per-step, not live log-tailing).
- An in-TUI text editor widget (no `tui-textarea` dep) — editing hands off to `$EDITOR` (DECISION-3).
- Live `distrobox enter` *rendered inside* a pane (we suspend the whole TUI for a real TTY; embedding a PTY is explicitly excluded).
- Package-removal / recreate prompts beyond what `core::apply` already enforces (TUI surfaces the existing exit-65 recreate path; see §6.5).

### 2.4 Assumptions (with risk-if-wrong)
| # | Assumption | Risk if wrong | Mitigation |
|---|---|---|---|
| A1 | `core::` functions are blocking but bounded enough that running them on a worker thread (not the render thread) keeps the UI at 60fps-ish. | A truly hung `distrobox` blocks the worker, not the UI; user can still see a spinner and quit. | Worker thread isolates blocking; a "Busy" model state disables conflicting input. |
| A2 | `DistroboxRunner` is `Send + Sync` (it is — see `runner.rs` trait bound), so an `Arc<dyn DistroboxRunner>` can be moved into the worker thread. | If a future runner were `!Send`, the worker model breaks. | Trait already requires `Send + Sync`; assert with a `fn _assert_send_sync()`. |
| A3 | Terminal restore via crossterm `LeaveAlternateScreen` + `disable_raw_mode` + a panic hook + `SIGINT` handling is sufficient on the target Linux hosts. | A panic in raw mode garbles the terminal. | RAII `TerminalGuard` + custom panic hook that restores before printing (see §5). |
| A4 | `cbox` with no args launching the TUI is desirable default UX. | Scripts that ran bare `cbox` expecting help/usage would now get a TUI. | DECISION-1: recommend `cbox` (no args, TTY-only) → TUI; non-TTY stdin/stdout → print help + exit 64. Justified below. |

## 3. ENTRY & FEATURE GATING (normative)

### 3.1 The feature flag — recommend `tui` **on by default**

Current `Cargo.toml`: `default = []`, `tui = ["dep:ratatui", "dep:crossterm"]`. **Recommendation (DECISION-2): flip to `default = ["tui"]`.** Justification:
- The product *is* "a cozy distrobox manager"; the cozy cockpit is a headline feature, not an opt-in extra. Shipping it off-by-default means the flagship UX is invisible to anyone who `cargo install cbox`.
- `make lint`/`make check` already build `--all-features`, so TUI code is compiled+clippy-gated regardless. Making it default removes the "works in CI, missing in release" trap.
- The CLI remains fully usable; the TUI adds two deps that are L1-locked and small relative to clap.
- **If the maintainer prefers a lean default binary**, keep `default = []` — the entry code below already handles "built without `tui`" gracefully. Either way the *code* is identical; only the `Cargo.toml` line changes. This is a 1-line decision, defaulted to "on", and does not block Vivi.

### 3.2 Dispatch — two entry paths, both in `cli::` and `main.rs`, both calling `tui::run`

The layering rule is `tui → core` only. `main.rs` and `cli/` may depend on `tui` (they already depend on everything). So the launch wiring lives at the `main.rs`/`cli/` boundary; `tui::run` itself only touches `core::`.

Add to `cli::Commands` a new variant:
```rust
/// Launch the cozy terminal cockpit.
Tui(tui_cmd::TuiArgs),   // cli/tui_cmd.rs — args struct only; NOT the TUI itself
```
And handle "no subcommand": make `Cli.command` an `Option<Commands>` **OR** keep it required and add a thin pre-parse. **Recommended:** change `#[command(subcommand)] pub command: Commands` → `pub command: Option<Commands>`, and in `dispatch`:
```text
match &cli.command {
    None            => launch_tui(cli, runner),          // `cbox`  (no args)
    Some(Tui(args)) => launch_tui_with(args, cli, runner),// `cbox tui`
    Some(other)     => /* existing P1/P2 dispatch */,
}
```

`launch_tui`:
1. **TTY guard.** If stdout or stdin is **not** a TTY (`std::io::IsTerminal`), do **not** start a TUI. For `cbox` (no args) on a non-TTY → print the clap help and exit `USAGE` (64). For `cbox tui` explicitly on a non-TTY → `CboxError::usage("cbox tui needs an interactive terminal")` (exit 64). This protects pipelines (`cbox | cat`) and `A4`.
2. **Build-without-`tui` path.** Behind `#[cfg(not(feature = "tui"))]`, `launch_tui` returns `CboxError::software("This build of cbox has no TUI. Rebuild with --features tui, or use the subcommands: cbox list, cbox create, …")` → exit 70. `cbox tui --help` still works (clap parses the arg struct regardless of feature).
3. **Feature on:** call `tui::run(RunConfig { backend_override, json:_ignored, … })`. `--json` is meaningless for a TUI → if `cli.json` is set with the TUI entry, exit 64 with `CboxError::usage("--json is not supported for the TUI")` (mirrors how `enter`/`edit` reject `--json`).

`tui::run` signature (the only public TUI entry; everything else is `pub(crate)` or private):
```rust
#[cfg(feature = "tui")]
pub fn run(cfg: TuiConfig, runner: Arc<dyn DistroboxRunner>) -> Result<(), CboxError>;
```
`main.rs` constructs `Arc::new(RealRunner)` and passes it in. Tests construct `Arc::new(MockRunner)`. **The runner is injected, never constructed inside `tui/`** — this is what makes the whole TUI mock-testable (G-NO-NET).

> Note: `RealRunner` is currently a unit struct passed by `&`. It is `Send + Sync` (no fields). Wrapping in `Arc` is additive and touches no P1 contract.

## 4. ARCHITECTURE & STATE MODEL (TEA / Elm)

### 4.1 Why TEA (Model–Message–update–view)
A strict **Model → Message → `update` (pure reducer) → `view` (pure render)** split is chosen because the spec's hard requirement is *"pure state-transition logic unit-testable without a terminal."* TEA gives exactly that: `update(&mut Model, Message) -> Vec<Effect>` is a pure function over data — no `ratatui`, no `crossterm`, no `DistroboxRunner` inside it. `view(&Model, &mut Frame)` is pure rendering. The only impure parts (read crossterm events, run effects on the worker, draw the frame) live in a thin `event_loop` shell that is **not** unit-tested (covered by the `#[ignore]` smoke check instead).

Rejected alternatives: an immediate-mode "fat update that calls the runner inline" (couples I/O into the reducer → not unit-testable, blocks render); a callback/observer mesh (hard to reason about, no single source of truth). TEA is the minimal architecture that satisfies the testability gate.

### 4.2 The `Effect` indirection (the key testability seam)

The reducer must not call the runner (that would block the render thread and couple I/O into the pure function). Instead `update` returns **`Effect`s** — declarative descriptions of side work. A separate executor turns an `Effect` into a `core::` call on the **worker thread**, and the worker posts the result back as a new `Message`.

```text
crossterm event ──► Message ──►  update(&mut Model, Msg) -> Vec<Effect>   (PURE, unit-tested)
                                      │
                                      ▼
                              Effect dispatched to worker thread
                                      │  (core::list_machine / create / rm / apply / up / inspect / doctor)
                                      ▼
                              Message::*Done(Result<…, CboxError>) ──► back into update
```

This means **every acceptance test is**: feed a `Message`, assert the resulting `Model` and the emitted `Vec<Effect>` — no terminal, no thread, no runner needed for the *reducer* tests. A second tier of tests wires a real `MockRunner` through the effect executor to prove the `core::` call shape (the G-NO-NET gate).

### 4.3 `Model` (the single source of truth)
```rust
pub struct Model {
    pub screen: Screen,            // active screen (enum below)
    pub boxes: Vec<BoxRow>,        // last successful list_machine result
    pub list_state: ListState,     // ratatui table selection (index + offset)
    pub selected: Option<usize>,   // index into boxes
    pub detail: Option<InspectResult>,   // inspect/detail payload
    pub wizard: Option<WizardState>,     // create wizard, when active
    pub confirm: Option<ConfirmState>,   // destroy modal, when active
    pub progress: Option<ProgressState>, // apply/up step progress, when active
    pub status: StatusLine,        // cozy status: Idle | Busy(msg) | Error(CboxError-derived) | Ok(msg)
    pub backend: Backend,          // resolved once at startup (Backend::detect_or_default)
    pub doctor: Option<DoctorResult>,    // populated on demand / on backend-unreachable
    pub should_quit: bool,         // event loop checks this each tick
    pub busy: bool,                // true while an Effect is in flight → gates conflicting input
    pub pending_enter: Option<EnterSpec>,   // set when an enter handoff must suspend the TUI (§5)
    pub pending_edit: Option<String>,       // boxfile path to open in $EDITOR (suspend) (§4.7/§5)
}

pub enum Screen { List, Detail, Wizard, ConfirmDestroy, Progress, DoctorPanel }
```
`StatusLine` is a small enum (not a raw `String`) so the view can color it and tests can assert the variant. Error text is derived from `CboxError::to_string()` (reuse, not re-author).

### 4.4 `Message` (every input the reducer understands)
```rust
pub enum Message {
    // ── input (from crossterm, normalized) ──
    Key(KeyEvent),        // raw key; update interprets per active screen
    Tick,                 // periodic redraw / spinner advance
    Resize(u16, u16),

    // ── effect completions (from the worker thread) ──
    ListLoaded(Result<Vec<BoxRow>, CboxError>),
    DetailLoaded(Result<InspectResult, CboxError>),
    CreateDone(Result<CreateOutcome, CboxError>),
    RmDone(Result<RmOutcome, CboxError>),
    ApplyDone(Result<ApplyOutcome, CboxError>),
    UpDone(Result<UpOutcome, CboxError>),
    DoctorDone(Result<DoctorResult, CboxError>),

    // ── terminal-handoff completions ──
    EnterReturned(Result<i32, CboxError>),  // posted after the suspended enter finishes
    EditReturned(Result<(), CboxError>),    // posted after $EDITOR returns
}
```
`KeyEvent` is crossterm's type, but the reducer only reads `.code`/`.modifiers`; tests construct `KeyEvent` values directly (crossterm is a normal dep here). To keep the reducer 100% terminal-free, **wrap** crossterm's key into a tiny internal `Key` enum at the event-loop boundary if Vivi prefers — recommended but optional; either way the reducer stays pure.

### 4.5 `Effect` (declarative side-work the worker executes)
```rust
pub enum Effect {
    LoadList,
    LoadDetail(InspectSpec),
    Create(CreateSpec),
    Rm(RmSpec),
    Apply(ApplySpec),
    Up(UpSpec),
    Doctor(DoctorSpec),
    SuspendAndEnter(EnterSpec),   // handled by the event-loop shell, not the worker (§5)
    SuspendAndEdit(String),       // boxfile path → $EDITOR (§5)
    Quit,
}
```
The reducer returns `Vec<Effect>`. The shell routes:
- data effects (`LoadList`, `Create`, `Rm`, `Apply`, `Up`, `LoadDetail`, `Doctor`) → worker thread → `core::*` → result `Message`;
- terminal effects (`SuspendAndEnter`, `SuspendAndEdit`) → run on the **main thread** (they need the real TTY) by tearing down + restoring the terminal around the call (§5);
- `Quit` → sets `should_quit`.

### 4.6 `update` (the pure reducer) — contract
```rust
pub fn update(model: &mut Model, msg: Message) -> Vec<Effect>;
```
Rules:
- Pure over `(Model, Message)`: no I/O, no thread spawn, no runner, no clock except an injected tick. Deterministic — same `(Model, Message)` ⇒ same `(Model', Vec<Effect>)`.
- `busy == true` ignores conflicting `Key` messages (e.g. you cannot start a second `Create` while one is running) but still honors `Tick`, `Resize`, and `Ctrl-C → Quit`.
- Effect completions clear `busy`, update the relevant `Model` field, and set `status` to `Ok(..)` or `Error(..)`.
- Key handling is **screen-dispatched**: `update` matches on `model.screen` first, then the key. This keeps each screen's bindings local and testable (§7).

### 4.7 `view` (pure render) — contract
```rust
pub fn view(model: &Model, frame: &mut ratatui::Frame);
```
- No mutation of `Model` except ratatui `ListState`/`TableState` scroll bookkeeping (kept in `Model` so it's deterministic).
- Renders per `model.screen`; overlays (`ConfirmDestroy`, error toast) draw on top of the list.
- A persistent bottom **status/help bar** shows context key hints (cozy: `↑↓ move · enter open · c create · d destroy · a apply · e edit · ? doctor · q quit`).

## 5. EVENT LOOP, CONCURRENCY & TERMINAL LIFECYCLE

### 5.1 The loop (the only impure shell — `tui/app.rs`)
```text
loop {
    terminal.draw(|f| view(&model, f))?;          // render current model
    if model.should_quit { break; }
    // block up to POLL_MS for either a crossterm event or a worker message:
    select:
        crossterm event ready  → msg = normalize(event)
        worker rx has a msg     → msg = that completion
        timeout (POLL_MS)       → msg = Message::Tick
    let effects = update(&mut model, msg);         // PURE
    for eff in effects { route(eff, &worker_tx, &mut model, &mut terminal)?; }
}
```
crossterm 0.28 has no native select over events+channel; implement with `crossterm::event::poll(POLL_MS)` then a non-blocking `worker_rx.try_recv()`. `POLL_MS ≈ 50ms` gives a responsive spinner without busy-spin.

### 5.2 Concurrency model — **synchronous core + one background worker thread**

The codebase is deliberately sync (v1.0 §11.4: no tokio). We keep that. But `core::create`/`apply`/`up` can run for many seconds (real `distrobox create` pulls an image). Blocking the render thread on that would freeze the UI. **Recommended (DECISION-4): a single long-lived worker thread + two channels.**

```text
main thread ──Effect──► worker_tx ──► [worker thread] ──► core::*(spec, store, &*runner) ──► result_tx ──► main
```
- One `std::thread` spawned at `run()` start, owning `Arc<dyn DistroboxRunner>` (clone of the injected Arc) and a `GuestStateStore`. It loops `recv()` on a job channel, runs the matching `core::` call **synchronously**, and sends a completion `Message` back.
- Only **one** job in flight at a time (the reducer's `busy` flag enforces this), so we need no job queue beyond the channel and no shared mutable `Model` across threads — the worker never touches `Model`, only specs in and results out. This sidesteps all the `Send`/locking complexity.
- `LoadDetail`/`LoadList`/`Doctor` are fast but still go through the worker for uniformity (and because even `inspect` shells out).
- **Testability:** the worker is a thin `match effect { … }` over `core::` calls. Tests bypass the thread entirely and call the executor function directly (`execute_effect(eff, &store, &*runner) -> Message`) on the test thread with a `MockRunner` — proving the exact `core::` call + spec without any threading flakiness. The thread is only exercised by the `#[ignore]` smoke test.

### 5.3 Terminal lifecycle & guaranteed restore (R-term)

- **Setup** (`run()` start): `enable_raw_mode()` + `execute!(stdout, EnterAlternateScreen, EnableMouseCapture?)`. Mouse capture is **off** in v3.0 (out of scope) — do not enable it.
- **RAII guard:** a `TerminalGuard` whose `Drop` runs `disable_raw_mode()` + `LeaveAlternateScreen`. So any early return / `?` / normal exit restores the terminal.
- **Panic hook:** install a `std::panic::set_hook` that first restores the terminal (raw off, leave alt-screen) **then** prints the panic. Without this, a panic in raw mode garbles the user's shell. The hook must be idempotent with `Drop` (guard checks a flag).
- **SIGINT (Ctrl-C):** crossterm delivers `Ctrl-C` as a `KeyEvent` while in raw mode (the default terminal SIGINT generation is suppressed by raw mode), so `update` maps `Ctrl-C → Effect::Quit`, exiting cleanly through the guard. We do **not** install a libc signal handler in v3.0 (raw mode already routes Ctrl-C to us). Document this as the chosen stance.

### 5.4 The `enter` handoff — alt-screen suspend (critical, §scope-5)

`enter` is interactive: it needs the **real** TTY (job control, signals, full-screen TUIs inside the box). We cannot run it on the worker thread, and we cannot run it while our own alt-screen + raw mode hold the terminal. Flow (handled on the **main thread**, triggered by `Effect::SuspendAndEnter(EnterSpec)`):

```text
1. terminal.show_cursor(); LeaveAlternateScreen; disable_raw_mode();   // hand the bare TTY back
2. let code = core::enter(&enter_spec, &*runner)?;                      // run_interactive → TTY pass-through
3. enable_raw_mode(); EnterAlternateScreen; terminal.clear();           // re-take the terminal
4. post Message::EnterReturned(Ok(code)) into the loop                  // reducer refreshes status + reloads list
```
- `core::enter` already calls `runner.run_interactive(inv)` with `RunMode::Interactive` — the TUI reuses it verbatim. **No new core code.**
- In the `Model`, the in-flight handoff is represented by `pending_enter: Option<EnterSpec>` set by the reducer and consumed by the shell; on return the shell posts `EnterReturned`. The reducer never blocks.
- **Testability (the named gate):** the reducer test asserts that pressing `Enter` on a selected running box yields `Effect::SuspendAndEnter(EnterSpec{ name: "<sel>", .. })` — i.e. the **interactive path is selected**, not the capture path. The effect-executor test (with `MockRunner`) asserts `core::enter` routes to `run_interactive` (the `RecordedCall.interactive == true`), reusing the existing AC-ENTER-style assertion. No real PTY is ever spawned in tests (R7 from v1.0 carries forward).

### 5.5 The Boxfile editor — hand off to `$EDITOR` via the same suspend (DECISION-3, recommended)

Same suspend/restore dance as `enter`, but the inner call is the existing `cli::edit`-style flow. **Problem:** the file-edit loop currently lives in `cli/edit.rs` (spawns `$EDITOR`, re-validates, prompts) — and `tui → cli` is forbidden. **Resolution:** lift the reusable pieces into `core::`:
- `core::resolve_boxfile_path` and `core::scaffold_boxfile` already exist in `core::` — the TUI calls them directly.
- The actual `$EDITOR` spawn + revalidate loop is small; **the TUI implements its own minimal version inside `tui/` (edit_handoff)** calling `boxfile::parse_and_validate` for revalidation. This duplicates ~30 lines of editor-spawn logic, which is acceptable and keeps the layering rule intact. (Do **not** add `tui-textarea`; see §8.) After a successful edit the reducer offers "apply now?" → `Effect::Apply`.

> Alternative considered & rejected: an embedded `tui-textarea` editor. It adds a dep, re-implements modal editing badly versus the user's real `$EDITOR`, and forks the edit UX from the CLI's `$EDITOR` story. Hand-off keeps one mental model. Flagged as DECISION-3 with default = `$EDITOR` hand-off.

## 6. SCREENS (normative per screen)

Each screen lists: **layout · key bindings · transitions · backing `core::` call**.

### 6.1 Box list — the home screen (`Screen::List`)
- **Layout:** full-width `ratatui::Table` — columns `NAME | STATUS | IMAGE | DOCKER | CBOX?` (maps directly to `BoxRow{ name, status, image, docker_mode, cbox_managed }`). Bottom status/help bar. Title `🍂 cbox — your cozy boxes`.
- **Backing call:** `core::list_machine(&backend, &*runner)` on entry and after any mutation → `Message::ListLoaded`.
- **Keys:** `↑/k`,`↓/j` move selection; `Enter` → if box **running**, `Effect::SuspendAndEnter` (§5.4); else (stopped) → open Detail; `i` → Detail (inspect); `c` → Create wizard; `d` → Confirm-destroy; `a` → Apply (uses box's resolved Boxfile); `u` → Up; `e` → edit Boxfile; `r` → refresh (`LoadList`); `?` → Doctor panel; `q`/`Esc` → quit.
- **Transitions:** `c→Wizard`, `d→ConfirmDestroy`, `i/Enter(stopped)→Detail`, `?→DoctorPanel`.
- **Empty state:** if `boxes.is_empty()` after a successful load → centered cozy panel: `No boxes yet. Press 'c' to create your first one. 📦` (§7 empty-state AC).

### 6.2 Inspect / detail panel (`Screen::Detail`)
- **Layout:** key/value panel from `InspectResult` — name, status, image, created, docker_mode, backend, id, boxfile_path, mounts (sub-list host→guest:mode), packages. Scrollable.
- **Backing call:** `core::inspect(&InspectSpec{ name, raw:false, backend }, &*runner)` → `Message::DetailLoaded`.
- **Keys:** `Esc`/`q` → back to List; `e` → edit this box's Boxfile; `a` → apply; `Enter` → enter (if running). `Esc` from Detail never quits the app (only List quits).

### 6.3 Create wizard (`Screen::Wizard`) → produces a `CreateSpec`
- **Layout:** a step indicator + one focused field per step. Steps:
  1. **name** (text input; validate against `^[a-zA-Z0-9][a-zA-Z0-9_.-]*$` — reuse the same regex concept as boxfile validation; reject empty);
  2. **image** (text input, prefilled with `registry.fedoraproject.org/fedora-toolbox:latest` from `CreateSpec::new` default);
  3. **packages** (free text, space-separated → `Vec<String>`);
  4. **docker-mode picker** (radio over `none | host | nested` → `DockerMode`);
  5. **confirm** (read-only summary; shows the resolved spec).
- **Backing call:** on confirm → build `CreateSpec` (via `CreateSpec::new(name, backend)` then set fields; `dry_run:false`), `Effect::Create(spec)` → worker → `core::create` → `Message::CreateDone`. On success: status `Ok("created <name>")`, refresh list, return to List.
- **Keys:** `Tab`/`Enter` next step; `Shift-Tab` prev; `←/→` toggle docker-mode radio; `Esc` cancel (with "discard wizard?" if any field is dirty); on confirm step `Enter` submits.
- **State:** `WizardState{ step: WizardStep, name, image, packages_raw, docker_mode, dirty }`. The reducer's wizard transitions are pure and unit-tested (§7).

### 6.4 Confirm-destroy modal (`Screen::ConfirmDestroy`)
- **Layout:** centered modal over the list: `Destroy "<name>"? Its $HOME is preserved unless you also remove it. [y]es / [n]o · [h] also remove $HOME`.
- **Backing call:** on confirm → build `RmSpec{ names:[name], force:true, rm_home:<h-toggle>, all:false, yes:true }`, `Effect::Rm` → `core::rm` → `Message::RmDone`. Reuses the **same** `RmSpec` the CLI builds.
- **Keys:** `y`/`Enter` confirm; `n`/`Esc` cancel; `h` toggles rm-home. The modal is the TUI analog of the CLI `-y` confirm gate.
- **Safety:** never auto-destroys; modal is mandatory (no "skip confirm" in v3.0). Mirrors CLI's confirm-unless-`-y`.

### 6.5 Apply / Up flow with provision step progress (`Screen::Progress`)
- **Layout:** header (`Applying "<name>"…` / `Bringing up "<name>"…`), a **diff summary** line (from `ApplyOutcome.diff` / `DiffResult.class` + changed fields), then a **per-step list** rendered from `ApplyOutcome.steps: Vec<ProvisionStepResult>` — each row: `[idx] type  STATUS  (duration_ms)` where STATUS ∈ `ran | skipped | copied | failed`, colored (skip=dim, ran/copied=green, failed=red). A footer summary from `ApplySummary{ ran, skipped, copied, failed }`.
- **Backing call:** `Effect::Apply(ApplySpec{ name, boxfile_path, force:false, redo:[], no_provision:false, recreate:false, yes:true, dry_run:false, backend })` → `core::apply(&spec, &store, &*runner)` → `Message::ApplyDone(Result<ApplyOutcome,_>)`. `Up` uses `core::up` with a `UpSpec` built from a `CreateSpec` (the box's Boxfile via `resolve_boxfile_path` + parse, or the wizard spec).
- **Progress granularity:** `core::apply` runs **synchronously and returns the full `ApplyOutcome`** (the engine does not stream per-step events today). So in v3.0 the progress screen shows a **spinner + "running provision…"** while busy, then renders the **complete** step list when `ApplyDone` arrives. Live per-step streaming would require changing the Phase-2 engine signature → **out of scope / would break the additive constraint.** (See §9 R5 + §12 DECISION-5 if live streaming is later wanted.)
- **Recreate path:** if `core::apply` returns `Err(DataErr{..})` because a recreate-class diff needs `--recreate` (the engine already returns exit-65 with a cozy message naming the forcing fields), the TUI surfaces that message in a modal: `These changes need a recreate (destroys the container; $HOME preserved). Recreate now? [y/n]`. On `y` → re-issue `Effect::Apply` with `recreate:true`. This reuses the existing engine behavior; no new core logic.
- **Keys:** during run, input is gated (`busy`); on completion `Enter`/`Esc` → back to List (refresh).

### 6.6 Boxfile editor (`Screen` transient — suspend, not a ratatui screen)
- Triggered by `e` from List/Detail → `Effect::SuspendAndEdit(path)` where `path = core::resolve_boxfile_path(name, &EditSpec{..}, &*runner)`. If the file is absent, scaffold with `core::scaffold_boxfile` first (TUI writes the file, same as `cli::edit` does).
- Suspend the alt-screen (§5.5), spawn `$EDITOR`, re-validate with `boxfile::parse_and_validate`, restore, post `Message::EditReturned`. On success, reducer sets status `Ok("saved Boxfile for <name>")` and offers apply.

### 6.7 Doctor panel (`Screen::DoctorPanel`)
- **Layout:** renders `DoctorResult` — distrobox present/version/supported, backend selected/podman/docker reachability, warnings list. Cozy ✓/✗ glyphs.
- **Backing call:** `core::doctor(&DoctorSpec{ backend_override }, &*runner)` → `Message::DoctorDone`.
- **Auto-trigger:** if `list_machine`/any data effect returns a backend-unreachable error (`CboxError::TempFail` / `Backend` with connection signals), the reducer auto-routes to `DoctorPanel` with a banner `Can't reach your backend — here's what cbox sees:` (lean-on-doctor per §scope-6).

## 7. ERROR / EMPTY / LOADING STATES

- **Loading / busy:** `busy == true` shows a spinner in the status bar (`Tick` advances a frame index in `Model`). Input gated except quit.
- **Empty:** no boxes → §6.1 cozy panel.
- **Error:** any effect `Err(CboxError)` → `status = StatusLine::Error(e.to_string())`, rendered as a dismissible bottom toast (cozy `error:` prefix matching the CLI's `eprintln!("error: {e}")`). `CboxError` is **reused**, never re-authored — the TUI calls `e.to_string()` and (optionally) `e.exit_code()` for severity coloring.
- **Backend unreachable:** auto-route to Doctor panel (§6.7), leaning on `core::doctor`.
- **Box-not-found / already-exists:** surfaced verbatim from `CboxError` (these strings are already cozy and crafted in `error.rs` / `backend_error`).

## 8. ARCHITECTURE FIT & DEPENDENCIES

### 8.1 Layering (preserved)
```
main.rs / cli/ ──(launch)──► tui::run ──► core:: ──► dbox/ (DistroboxRunner seam) ──► [distrobox/podman/docker]
                                                  └► boxfile/ (parse_and_validate for editor revalidate)
```
`tui/` depends on `core/`, `dbox/` (for the runner trait + `Backend`), `boxfile/` (revalidate), and `error/`. **`tui/` never imports `cli/`.** The launch wiring lives in `main.rs`/`cli/`, which already sit above everything.

### 8.2 New files / modules under `src/tui/`
| File | Contents |
|---|---|
| `src/tui/mod.rs` | `pub fn run(cfg, runner)`; `TuiConfig`; re-exports; `#[cfg(not(feature="tui"))]` no-op stub kept for non-feature builds. |
| `src/tui/model.rs` | `Model`, `Screen`, `StatusLine`, `WizardState`, `WizardStep`, `ConfirmState`, `ProgressState`, `ListState` bookkeeping. **Pure data.** |
| `src/tui/message.rs` | `Message`, internal `Key` wrapper (optional). |
| `src/tui/effect.rs` | `Effect` enum + `execute_effect(eff, &store, &*runner) -> Message` (the worker's body; unit-testable directly). |
| `src/tui/update.rs` | `pub fn update(&mut Model, Message) -> Vec<Effect>` — the pure reducer, screen-dispatched. **Most-tested file.** |
| `src/tui/view.rs` | `pub fn view(&Model, &mut Frame)` + per-screen render fns. Not unit-tested (smoke only). |
| `src/tui/app.rs` | The impure shell: `TerminalGuard` (RAII), panic hook, event loop, worker-thread spawn, suspend/restore for enter+edit. Not unit-tested. |
| `src/cli/tui_cmd.rs` | `TuiArgs` clap struct (so `cbox tui --help` works regardless of feature). |

New types are all `pub(crate)` except `tui::run`, `TuiConfig`, and `TuiArgs`.

### 8.3 New types added outside `tui/`
- `cli::Commands::Tui(TuiArgs)` variant + `Cli.command: Option<Commands>` (additive enum variant + optionalizing the field). This is the **only** P1-surface touch and it is additive (no existing variant changes; existing dispatch arms unchanged).
- `main.rs`: wrap runner in `Arc`. Additive.
- No `core::`, `dbox::`, `boxfile::`, `error::` changes. **The P1/P2 contract is untouched.**

### 8.4 Dependencies — minimal, no new crates
| Crate | Status | Note |
|---|---|---|
| `ratatui` 0.28 | already declared (optional, `tui` feature) | widgets, layout, frame. |
| `crossterm` 0.28 | already declared (optional, `tui` feature) | raw mode, alt-screen, event poll, key events. |
| `std::io::IsTerminal` | std (Rust ≥1.70; host 1.96) | TTY guard. No dep. |
| `std::thread` + `std::sync::mpsc` | std | worker thread + channels. No dep. |
| `std::sync::Arc` | std | share the runner into the worker. No dep. |

**Explicitly avoided:** `tui-textarea` (editor hands off to `$EDITOR`, DECISION-3); `tokio`/async (locked-out by v1.0 §11.4; the worker thread covers blocking without async); any signal-handling crate (raw mode routes Ctrl-C to us, §5.3).

## 11. SPECTRA CYCLE NOTES

- **G3 evaluator-optimizer** (complexity ≥7 + ambiguous) was *considered*. Two architectures were weighed (TEA-with-Effect vs immediate-mode-fat-update); TEA-with-Effect dominates on the hard requirement (testable pure reducer). The concurrency choice (worker thread vs blocking-render vs async) was decided by the locked "sync core, no tokio" constraint + the "render must not block" requirement → worker thread is the only fit. No further refinement cycle was needed; confidence held ≥0.85 single-pass.
- **Confidence: 0.87.** The −0.13 is concentrated in: live-progress streaming being deferred (R5/DECISION-5), child-interrupt on Ctrl-C deferred (R2/DECISION-6), and the `Cli.command` optionalizing (R7) — all flagged for maintainer sign-off in §12, each with a safe recommended default so Vivi is unblocked.
