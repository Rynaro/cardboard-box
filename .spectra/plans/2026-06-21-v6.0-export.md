# cbox v6.0 — `cbox export` (distrobox-export integration)

**Spec ID:** `2026-06-21-v6.0-export`
**Status:** DECISION-READY · VALIDATE (confidence 0.84) — 5 scope decisions go to the maintainer first
**Intent:** `CHANGE` (extend existing crate) · Complexity 8/12 (extended-depth)
**Author:** SPECTRA · **Executor:** Vivi (Builder/Reasoner)
**Build:** containerized only — `make test` / `make check`, never bare `cargo` on host.

> Closes the documented gap: README.md:561 ("No distrobox-export yet … future feature"),
> docs/specs/v1.0-cli-lifecycle.md:72 (deferred "`distrobox-export` app integration").
> `distrobox-export` is ALREADY recognized as a distrobox-injected binary by the mount-diff
> engine (`src/core/diff.rs:302`) — so the host knows the helper exists in the box; what is
> missing is a cbox surface that *invokes* it.
>
> **Mechanism (grounded, not assumed):** `distrobox-export` runs INSIDE the box. cbox runs on
> the HOST. So every export is an exec into the box:
> `distrobox enter --name <box> -- distrobox-export <subflags>`
> — structurally identical to the existing provision-shell argv
> (`build_provision_shell_argv` / `build_state_read_argv`, `argv.rs:219,260`) and the
> `enter` path (`core::enter`, `mod.rs:557`). It is fully mock-testable via `MockRunner`'s
> `RecordedCall { program, args, env }` (`mock.rs:11`) — zero real distrobox, zero host writes.

## 0. Scope

### In scope (v6.0 — pending the §3 decision rulings)
- A `cbox export` command surface that wraps `distrobox-export` by exec'ing into the box.
- An MVP capability subset: `--app`, `--bin`, `--service`, `--delete`, `--list-apps`/`--list-binaries`.
- Human-readable default output + `--json` machine mode (consistent with the rest of cbox).
- Precondition checks (box exists; box reachable) mapped onto the existing exit-code taxonomy.
- Failure mapping for the three realistic failure shapes: box not running, `distrobox-export`
  missing in the box, app/bin not found.
- Documentation: README section demonstrating export round-trips; remove the "No
  distrobox-export yet" disclaimer; update v1.0 deferred list.

### Out of scope / Deferred (v6.0)
- A declarative `[[exports]]` Boxfile section participating in diff/apply (DECISION-1 — the
  RECOMMENDED v6.0 cut defers this; rationale + a forward-compatible seam in §3.1, §8).
- `--export-label`, `--extra-flags`, `--sudo` pass-through (DECISION-2 — deferred; §3.2).
- Host→box direction or any non-`distrobox-export` mechanism. cbox export is strictly the
  box→host surfacing direction `distrobox-export` itself implements (DECISION-3).
- Listing/cleaning ORPHANED host launchers whose box was removed (a host-side GC pass). Noted
  as a future `cbox export prune` in §11 RR-3.
- TUI export panel. The CLI mechanism is the seam; a TUI affordance is a later bundle.

### Assumptions (risk-if-wrong)
- **A1** `distrobox-export` exists and is on PATH inside a distrobox-created box (it is
  injected at `/usr/bin/distrobox-export` — confirmed by `diff.rs:302`'s injected-binary list).
  Risk-if-wrong: an exec returns "command not found"; we map that to a clear exit 70 with an
  actionable message (§2.4 F2), not a panic. The box must have been created by distrobox
  (cbox always creates via distrobox), so the binary is present by construction for cbox boxes.
- **A2** The box must be RUNNING for `distrobox enter -- …` to succeed; distrobox
  auto-starts a stopped box on `enter`. Risk-if-wrong: distrobox itself handles the
  cold-start; if start fails the backend error surfaces as exit 125 (existing
  `backend_error` path). We do NOT add our own start orchestration in v6.0 — we rely on
  distrobox's enter-auto-start, exactly as `core::enter` and the provision walk already do.
- **A3** `distrobox-export`'s flags are stable across distrobox ≥ 1.6 (the documented cbox
  floor, README:553): `--app`, `--bin`/`--export-path`, `--service`, `--delete`,
  `--list-apps`, `--list-binaries`. Risk-if-wrong: a distrobox version skew changes a flag;
  cbox passes the flags through verbatim, so a skew surfaces as `distrobox-export`'s own
  non-zero exit + stderr (exit 125, stderr-tail shown), not a silent miscompile.
- **A4** Export side effects (host `.desktop` files, wrapper scripts) land on the HOST
  filesystem owned by the user. cbox does not need to read or verify them; success is the
  exec exit code + `distrobox-export`'s own stdout. Risk-if-wrong: a partial export that
  exits 0 but wrote nothing — out of cbox's control and out of v6.0 scope (we report what
  the helper reported).

## 1. Command surface

### 1.1 Narrative

A single subcommand `Export(export::ExportArgs)` with a small, flag-driven surface that
mirrors `distrobox-export`'s own shape so a user who knows the tool transfers their
knowledge directly (and a user who does not gets `cbox export --help`):

```
cbox export <BOX> --app <appname>                     # surface a GUI app as a host .desktop
cbox export <BOX> --bin <guest/path> [--to <hostdir>] # surface a binary as a host wrapper
cbox export <BOX> --service <name>                    # surface a systemd service
cbox export <BOX> --app <appname>   --delete          # remove a previously-exported app
cbox export <BOX> --bin <guest/path> --delete         # remove a previously-exported binary
cbox export <BOX> --service <name>  --delete          # remove a previously-exported service
cbox export <BOX> --list-apps                          # list currently-exported apps
cbox export <BOX> --list-bins                          # list currently-exported binaries
```

- `<BOX>` is positional and REQUIRED — there is no implicit "current box"; export always
  targets a named box (matches `enter`/`apply`/`inspect`, which all take a NAME).
- Exactly ONE export *target* mode must be given per invocation: one of `--app`, `--bin`,
  `--service`, `--list-apps`, `--list-bins`. Zero or two+ target modes → exit 64 usage error
  (clap `ArgGroup` `required = true, multiple = false`).
- `--delete` is a MODIFIER on `--app`/`--bin`/`--service` (not valid with the list modes).
  Combining `--delete` with a list mode → exit 64.
- `--to <hostdir>` maps to `distrobox-export`'s `--export-path` and is ONLY meaningful with
  `--bin`. With any other mode → exit 64. When omitted with `--bin`, cbox passes NO
  `--export-path` and lets `distrobox-export` use its own default (it requires
  `--export-path` for `--bin`; if the user omits it, `distrobox-export` errors — we surface
  that as F3, rather than inventing a default host dir of our own). See DECISION-2 note.

### 1.2 clap representation (new file `src/cli/export.rs`)

```rust
#[derive(Args, Debug)]
pub struct ExportArgs {
    /// Box name.
    #[arg(value_name = "BOX")]
    pub box_name: String,

    /// Export a graphical app (creates a host .desktop launcher).
    #[arg(long, value_name = "APPNAME", group = "target")]
    pub app: Option<String>,

    /// Export a binary as a host wrapper script.
    #[arg(long, value_name = "GUEST_PATH", group = "target")]
    pub bin: Option<String>,

    /// Export a systemd service.
    #[arg(long, value_name = "NAME", group = "target")]
    pub service: Option<String>,

    /// List currently-exported apps.
    #[arg(long, group = "target")]
    pub list_apps: bool,

    /// List currently-exported binaries.
    #[arg(long, group = "target")]
    pub list_bins: bool,

    /// Host directory for an exported binary (maps to distrobox-export --export-path).
    /// Only valid with --bin.
    #[arg(long, value_name = "HOSTDIR")]
    pub to: Option<String>,

    /// Remove a previously-created export (combine with --app/--bin/--service).
    #[arg(long)]
    pub delete: bool,
}
```

`#[group(required = true, multiple = false)] target` (declared on the `ExportArgs` struct via
`#[command(group = ArgGroup::new("target").required(true).multiple(false))]`) makes clap
enforce "exactly one target mode" and emit the exit-64 usage error for free — matching how
the rest of cbox leans on clap for arg-shape errors.

### 1.3 The exec contract — argv shape (new builders in `src/dbox/argv.rs`)

Every export is `distrobox enter --name <BOX> -- distrobox-export <flags>`, run in
`RunMode::Capture` (we want the helper's stdout/stderr + exit code, not an interactive TTY).

```rust
/// distrobox enter --name <NAME> -- distrobox-export --app <APP> [--delete]
pub fn build_export_app_argv(name: &str, app: &str, delete: bool) -> Vec<String>;

/// distrobox enter --name <NAME> -- distrobox-export --bin <PATH> --export-path <DIR> [--delete]
/// (--export-path omitted when `to` is None; --delete may omit --export-path too)
pub fn build_export_bin_argv(name: &str, bin: &str, to: Option<&str>, delete: bool) -> Vec<String>;

/// distrobox enter --name <NAME> -- distrobox-export --service <NAME> [--delete]
pub fn build_export_service_argv(name: &str, service: &str, delete: bool) -> Vec<String>;

/// distrobox enter --name <NAME> -- distrobox-export --list-apps
pub fn build_export_list_apps_argv(name: &str) -> Vec<String>;

/// distrobox enter --name <NAME> -- distrobox-export --list-binaries
pub fn build_export_list_bins_argv(name: &str) -> Vec<String>;
```

All five return a `Vec<String>` of the form
`["enter", "--name", <BOX>, "--", "distrobox-export", <subflags…>]` — exactly the
`build_provision_shell_argv` shape (`argv.rs:219`) with `distrobox-export` substituted for
`sh -c`. Each gets in-module `#[cfg(test)]` unit tests asserting the exact token vector
(deterministic order: target flag, then value, then `--delete` last for the mutating forms).

> **Why a builder per mode, not one mega-builder:** matches the existing `argv.rs` idiom
> (one `build_*_argv` per concrete distrobox call shape — `build_create_argv`,
> `build_enter_argv`, `build_provision_shell_argv`, `build_copy_argv`). Five tiny pure
> functions are trivially unit-testable and keep `core::export` a thin dispatcher.

## 2. Core orchestration (`src/core/mod.rs` — new `pub fn export`)

### 2.1 Spec type (`src/core/spec.rs`)

```rust
/// What the user asked export to do (exactly one target).
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum ExportTarget {
    App { name: String },
    Bin { path: String, to: Option<String> },
    Service { name: String },
    ListApps,
    ListBins,
}

/// Spec for `cbox export`.
#[derive(Debug, Clone)]
pub struct ExportSpec {
    pub box_name: String,
    pub target: ExportTarget,
    pub delete: bool,        // only meaningful for App/Bin/Service; validated in cli/export.rs
    pub backend: Backend,    // pins distrobox to the right engine (same as enter/apply)
    pub dry_run: bool,
}

/// Outcome of `cbox export` (serialized for --json).
#[derive(Debug, Clone, serde::Serialize)]
pub struct ExportOutcome {
    pub ok: bool,
    pub action: String,      // "export" | "export-delete" | "export-list"
    pub box_name: String,
    pub mode: String,        // "app" | "bin" | "service" | "list-apps" | "list-bins"
    pub target: Option<String>,    // app/bin/service name; None for list modes
    pub deleted: bool,
    /// For list modes: the parsed entries distrobox-export reported (one per line).
    pub entries: Vec<String>,
    /// distrobox-export's own stdout headline (human echo / provenance). Never includes secrets.
    pub detail: String,
}
```

### 2.2 The core function

```rust
pub fn export(spec: &ExportSpec, runner: &dyn DistroboxRunner)
    -> Result<ExportOutcome, CboxError>;
```

Flow (single, linear — no diff/state, this is imperative per DECISION-1):

1. **Resolve backend** via the existing `resolve_backend(&spec.box_name, override, runner)`
   (`mod.rs:348`) — already done by the CLI layer; `spec.backend` is pre-pinned.
2. **Existence precondition.** Reuse `inspect()` (`mod.rs:567`) to confirm the box exists.
   `Err` with exit code `UNAVAILABLE (69)` → re-raise as `box_not_found` (exit 69), matching
   `apply`'s precondition pattern (`mod.rs:1135-1141`). This gives the user the canonical
   "No box named X. Create it with: cbox create X" message for free.
   - We do NOT hard-fail on a STOPPED box (A2): distrobox auto-starts on `enter`. We only
     assert existence, not running-state. (A stopped box that fails to start surfaces as the
     backend's own exit-125 error from the enter call.)
3. **Build argv** via the §1.3 builder selected by `spec.target` + `spec.delete`.
4. **Run** `Invocation::new("distrobox", args, Capture).with_env(vec![backend.dbx_env()])`
   (the `dbx_env()` backend-pin is the same one `core::enter` uses, `mod.rs:560`).
   - `dry_run`: if `spec.dry_run`, build the `Invocation` with `RunMode::DryRun` so the
     runner returns the would-be argv as stdout without spawning (consistent with the global
     `--dry-run` contract, `runner.rs:13`). Print argv, return `ok:true`, action unchanged.
5. **Map the result** (§2.3 / §2.4).

### 2.3 Success mapping

- **Mutating modes (`--app`/`--bin`/`--service`, with/without `--delete`):** exit 0 from
  `distrobox-export` → `ExportOutcome { ok:true, action: if delete {"export-delete"} else {"export"},
  mode, target: Some(name), deleted: delete, entries: [], detail: <stdout headline> }`.
- **List modes (`--list-apps`/`--list-bins`):** exit 0 → parse stdout into `entries`
  (split on `\n`, trim, drop empties). `action:"export-list"`, `target:None`.

### 2.4 Failure mapping (onto the existing taxonomy — DECISION-4)

The core principle: cbox owns the PRECONDITION errors (clear, cbox-shaped); it passes
through the helper's RUNTIME errors with the existing `backend_error` machinery so the user
sees `distrobox-export`'s own diagnostic.

| # | Failure | Detection | Maps to | Exit |
|---|---------|-----------|---------|------|
| F1 | Box does not exist | `inspect()` → UNAVAILABLE | `CboxError::box_not_found` | 69 |
| F2 | `distrobox-export` not found in box | exec exit ≠ 0 AND stderr contains `distrobox-export` + (`not found`/`No such file`/`command not found`) | `CboxError::software` w/ actionable msg | 70 |
| F3 | App/bin/service not found, or `--export-path` required-but-missing, or any other helper error | exec exit ≠ 0 (catch-all) | `CboxError::backend_error(code, stderr, argv)` (existing cozy patterns + stderr tail) | 125 |
| F4 | Backend (podman/docker) unreachable / box won't start | `RunnerError` or backend `refused`/`cannot connect` stderr | existing `backend_error` / `RunnerError` mapping | 125 / via RunnerError |
| F5 | Bad usage (no target, two targets, `--to` w/o `--bin`, `--delete` w/ list mode) | clap `ArgGroup` + cli/export.rs guards | `CboxError::usage` | 64 |

- **F2 message (the one cbox-authored runtime message):**
  `"distrobox-export isn't available inside \"<BOX>\". This box may not be a distrobox-managed box, or its image lacks the distrobox integration. See:  cbox doctor"`
  (exit 70). F2 is detected by stderr-substring BEFORE the generic F3 catch-all, so the
  user gets the specific guidance rather than a raw helper dump.
- **F3** deliberately reuses `backend_error` (`error.rs:135`) so the existing cozy
  "image not found / already exists / refused" pattern-matching applies and the stderr tail
  is shown. distrobox-export's own "Trying to export… not found" lines land in the tail.
- No NEW exit codes are introduced. The taxonomy (64/65/69/70/74/75/125) is reused exactly
  — 65/74/75 simply don't arise on this path (no Boxfile data-parse, no keyring), which is
  correct and documented.

## 3. Scope decisions (→ maintainer ruling before execution)

Each decision is a genuine fork. Recommendation + rationale + assumption-with-risk-if-wrong.

### DECISION-1 — Declarative vs imperative · **RECOMMEND: imperative-only in v6.0; reserve a forward-compatible `[[exports]]` seam, do NOT implement diff/apply for it yet.**

- **Options.** (a) Imperative command only — `cbox export <box> --app foo`. (b) Also a
  declarative `[[exports]]` Boxfile section that participates in the diff/apply/idempotency
  engine like `[secrets]`/`[[mounts]]`.
- **Recommendation:** (a) for v6.0. Ship the imperative command. In the schema work,
  RESERVE the top-level key `exports` in the `known_top_level` allowlist
  (`boxfile/mod.rs:15`) so a future Boxfile that declares `[[exports]]` does NOT warn — but
  do NOT add a serde struct, validation, diff classification, or apply wiring in v6.0.
- **Rationale.**
  - *Scope/complexity.* The declarative path is genuinely hard to make idempotent and
    truthful (see DECISION-5): `distrobox-export` writes HOST files cbox cannot reliably
    diff from a guest exec, and re-running an export is already (per the helper) idempotent
    at the helper level, so a cbox-side diff would either duplicate the helper's idempotency
    or risk the box.home blind-spot the v5.0 spec explicitly warned against (a declared-but-
    never-truthfully-checked field). Doing it wrong is worse than not doing it.
  - *Consistency cost is bounded.* cbox's declarative model is strong, and a future
    `[[exports]]` is a natural fit — but the imperative command is the right FIRST cut: it
    delivers the user value (surface an app NOW) with a small, fully-mock-testable surface,
    and it gives us real usage data to design the declarative shape correctly later.
  - *Cheap forward-compat.* Reserving the `exports` allowlist key (one array element) costs
    nothing and prevents a future warning regression; it signals intent without committing
    to the engine.
- **Assumption / risk-if-wrong.** Assumes users accept "exports are an imperative action,
  re-run on a fresh box" for now. Risk-if-wrong: users expect `cbox up` to re-surface their
  apps automatically after a recreate; if that demand is strong, v6.1 adds the declarative
  section. Mitigation: the reserved seam + this spec's §8 sketch make v6.1 a clean additive
  follow-up, not a rework.

### DECISION-2 — Command surface (v6.0 subset) · **RECOMMEND: `--app`, `--bin` (+`--to`), `--service`, `--delete`, `--list-apps`, `--list-bins`. DEFER `--export-label`, `--extra-flags`, `--sudo`.**

- **Recommendation:** Ship the five target modes + the `--delete` modifier + `--to` (the
  `--export-path` for `--bin`, which `distrobox-export` REQUIRES for binaries, so it is not
  optional sugar — it is needed for `--bin` to work at all). Defer the three "supporting
  flags."
- **Rationale.**
  - `--app`/`--bin`/`--service`/`--delete`/`--list-*` are the *primary verbs* — they cover
    the whole "surface a resource / remove it / see what's surfaced" loop. That is a
    coherent MVP.
  - `--export-label`, `--extra-flags`, `--sudo` are *refinements*: `--export-label`
    renames a launcher, `--extra-flags` injects raw flags into the generated wrapper,
    `--sudo` exports as root. Each adds surface and test matrix for a minority case. Defer
    until the core loop is proven. `--extra-flags` in particular is a passthrough-of-raw-
    strings risk (escaping/quoting) better designed deliberately later.
  - `--to`/`--export-path` is NOT deferrable: without it `distrobox-export --bin` errors
    (F3). Including it keeps `--bin` actually usable.
- **Assumption / risk-if-wrong.** Assumes the deferred flags are minority needs.
  Risk-if-wrong: a user needs `--sudo` for a root-owned binary in v6.0 — they can still run
  `cbox enter <box> -- distrobox-export --bin … --sudo` manually (documented escape hatch),
  and v6.1 adds the flag as an additive option. Low risk; no rework (additive).

### DECISION-3 — Direction & semantics · **RECOMMEND: confirm box→host only; `--delete` and list modes operate on the box's view of its own exports.**

- **Recommendation:** `cbox export` is exclusively the box→host *surfacing* direction that
  `distrobox-export` itself implements: it makes a container app/bin/service appear on the
  HOST (a host `.desktop`, a host wrapper script, a host-visible service). There is no
  host→box mode (that is `distrobox-host-exec` territory, out of scope).
  - `--delete` removes a previously-created host artifact for the named app/bin/service —
    cbox passes `--delete` through; `distrobox-export` owns the removal of the host file it
    created. cbox does not independently track or delete host files.
  - `--list-apps`/`--list-bins` reflect what `distrobox-export` reports as currently
    exported *from that box* (the helper's own listing, run inside the box). cbox parses and
    presents that list; it does not enumerate the host filesystem itself.
- **Rationale.** This is the only honest semantics: the helper does the work, cbox is the
  ergonomic host-side front door + precondition/error shaping. Claiming cbox "manages host
  launchers" would over-promise (it does not own a host-side registry in v6.0 — see RR-3).
- **Assumption / risk-if-wrong.** Assumes the maintainer agrees cbox should NOT build a
  host-side export registry/GC in v6.0. Risk-if-wrong: orphaned launchers after `cbox rm`
  (the box is gone but its host `.desktop` remains). Documented honestly (RR-3) and flagged
  as a possible future `cbox export prune`. Not a v6.0 defect — it matches `distrobox-export`'s
  own behavior.

### DECISION-4 — Output & exit codes · **RECOMMEND: human default + `--json`; failures map onto the EXISTING taxonomy (no new codes).**

- **Recommendation (the §2.4 table).** Human-readable default; `--json` emits `ExportOutcome`.
  - **Box not found → 69** (`box_not_found`, same as `apply`/`inspect`).
  - **`distrobox-export` missing in box → 70** (`software`, the one cbox-authored runtime msg).
  - **App/bin/service not found OR helper error (incl. `--export-path` missing) → 125**
    (`backend_error`, helper's own stderr-tail shown — the wrapped-non-zero class).
  - **Bad usage (no/two targets, `--to` w/o `--bin`, `--delete` w/ list) → 64** (`usage`).
  - **Backend unreachable / won't start → 125 or via `RunnerError`** (existing mapping).
- **`--json` shape (mutating):**
  `{"ok":true,"action":"export","box":"dev","mode":"app","target":"firefox","deleted":false,"entries":[],"detail":"…"}`
- **`--json` shape (list):**
  `{"ok":true,"action":"export-list","box":"dev","mode":"list-apps","target":null,"deleted":false,"entries":["firefox","gimp"],"detail":""}`
- **Rationale.** Reusing the taxonomy keeps the contract uniform — a script that already
  handles cbox exit codes needs no new branches. 69 for "no such box" is the established
  meaning (`exit::UNAVAILABLE`); 70 for "the tool you need is broken/absent"; 125 for
  "the wrapped command failed" (`exit::BACKEND_NONZERO`). This is the same split `apply`
  already uses (precondition vs wrapped-failure).
- **Assumption / risk-if-wrong.** Assumes 70-vs-125 for F2-vs-F3 is the right cut. Risk: a
  reviewer prefers F2 (missing helper) also be 125 (it IS a non-zero exec). We chose 70
  because it is a SOFTWARE/environment problem the user must fix (wrong image), distinct
  from "the app name was wrong" (a data/usage-ish runtime failure → 125). If the maintainer
  prefers uniform 125, that is a one-line change with no structural impact.

### DECISION-5 — Idempotency (only relevant IF DECISION-1 chooses declarative) · **RECOMMEND: not applicable in the recommended (imperative) cut; documented re-run semantics instead.**

- **Recommendation (under the recommended imperative cut):** No cbox-side idempotency engine.
  `distrobox-export` is itself idempotent for the common cases (re-exporting the same app is
  a no-op / overwrite at the helper level; `--delete` of an absent export is benign). cbox
  surfaces the helper's exit code. We document: "re-running `cbox export … --app foo` is
  safe; it re-asserts the host launcher."
- **IF the maintainer overrides DECISION-1 to declarative,** the idempotency design would be:
  - A value-free `exports_fingerprint` over the sorted `[[exports]]` entries
    (`{kind, target, to?}`), stored in `ProvisionState` as a new `#[serde(default)]` field
    (exact pattern as v5.0's `env_secret_fingerprint`, `state_store.rs:29`).
  - It MUST be actually checked AND read back — NOT assumed-unchanged (the explicit
    box.home / boxfile_sha anti-pattern guard from the v5.0 spec, §7.2). A new `DiffField{
    field:"exports", class:"Incremental"}` (exports are additive host artifacts; adding one
    re-runs the export, removing one runs `--delete`; never a Recreate).
  - The HARD honesty caveat: cbox cannot verify the host `.desktop`/wrapper actually still
    exists from a guest exec, so the fingerprint tracks DECLARED intent, not host reality.
    This truthfulness gap is exactly why DECISION-1 recommends deferring it.
- **Assumption / risk-if-wrong.** Assumes the helper's own idempotency is sufficient for
  v6.0. Risk-if-wrong: a user re-runs and gets a duplicate launcher — not observed in
  `distrobox-export`'s documented behavior (it overwrites), so low risk; if it bites,
  DECISION-1 → declarative in v6.1 with the seam above.

## 4. Output rendering (`src/cli/export.rs` + `src/cli/output.rs`)

- **Human, mutating success:** cozy success line via `ctx.success(...)`, e.g.
  `Exported app "firefox" from "dev" to your host menu.` /
  `Removed exported app "firefox" for "dev".` Echo `distrobox-export`'s headline under it in
  non-quiet mode if non-empty.
- **Human, list success:** one entry per line (matches `secret list`'s names-only line
  format, `cli/secret.rs:195`); empty → `No apps exported from "dev".`
- **`--json`:** `ctx.print_json(&outcome)` (the established `OutputCtx` JSON path used by
  every other command).
- **`-v` (verbose argv echo):** the existing `OutputCtx` verbose-argv path prints the
  `distrobox enter … -- distrobox-export …` argv. There are NO secrets on this path, so no
  redaction concern (unlike v5.0). Document that `-v` shows the full export argv.

## 7. Acceptance suite (GIVEN / WHEN / THEN) — all mock-driven

All via `MockRunner` (`RecordedCall { program, args, env }`) + `assert_cmd` for the CLI
surface. Zero real distrobox, zero host writes (G-MOCK / G-NO-NET preserved).

**AC-EXP-1 — `--app` builds the correct exec argv (mutating, success)**
GIVEN a `MockRunner` whose default response is `ok("")` and a box "dev" that `inspect`
reports as existing
WHEN `core::export` runs with `ExportTarget::App{name:"firefox"}`, `delete:false`
THEN the recorded call is `program == "distrobox"` with
`args == ["enter","--name","dev","--","distrobox-export","--app","firefox"]`
AND `ExportOutcome { ok:true, action:"export", mode:"app", target:Some("firefox"), deleted:false }`.

**AC-EXP-2 — `--app --delete` appends `--delete`**
GIVEN the same setup
WHEN `core::export` runs with `App{firefox}`, `delete:true`
THEN recorded `args` end with `"--app","firefox","--delete"`
AND `action == "export-delete"`, `deleted == true`.

**AC-EXP-3 — `--bin --to` maps to `--export-path`**
WHEN `core::export` runs with `Bin{path:"/usr/bin/htop", to:Some("/home/u/.local/bin")}`
THEN recorded `args` ==
`["enter","--name","dev","--","distrobox-export","--bin","/usr/bin/htop","--export-path","/home/u/.local/bin"]`.

**AC-EXP-4 — `--bin` without `--to` passes no `--export-path`**
WHEN `Bin{path:"/usr/bin/htop", to:None}`
THEN recorded `args` contain `"--bin","/usr/bin/htop"` and do NOT contain `"--export-path"`.

**AC-EXP-5 — `--service` builds service argv**
WHEN `Service{name:"nginx"}`, delete:false
THEN recorded `args` == `[…,"distrobox-export","--service","nginx"]`.

**AC-EXP-6 — `--list-apps` parses entries**
GIVEN a `MockRunner` matcher returning stdout `"firefox\ngimp\n"`
WHEN `core::export` runs with `ListApps`
THEN recorded `args` end with `"distrobox-export","--list-apps"`
AND `ExportOutcome { action:"export-list", mode:"list-apps", entries:["firefox","gimp"], target:None }`.

**AC-EXP-7 — `--list-bins` parses entries** (mirror of AC-EXP-6 with `--list-binaries`).

**AC-EXP-8 — box does not exist → exit 69, no export attempted (F1)**
GIVEN a `MockRunner` whose `inspect` returns the not-found shape (`[]`)
WHEN `core::export` runs
THEN it returns `CboxError` with `exit_code() == 69`
AND the recorded calls contain the `inspect` call but NO `distrobox-export` call
(`!calls.iter().any(|c| c.args.contains("distrobox-export"))`).

**AC-EXP-9 — `distrobox-export` missing → exit 70 with cbox-authored message (F2)**
GIVEN a box that exists, and the exec call returns `err(127, "distrobox-export: command not found")`
WHEN `core::export` runs with `App{firefox}`
THEN `exit_code() == 70`
AND the message contains `distrobox-export isn't available` and `cbox doctor`.

**AC-EXP-10 — app not found → exit 125 with helper stderr tail (F3)**
GIVEN the exec returns `err(1, "Trying to export … application firefox not found")`
WHEN `core::export` runs
THEN `exit_code() == 125`
AND the error's stderr-tail contains `not found` (the helper's own message surfaced).

**AC-EXP-11 — `--bin` with missing `--export-path` surfaces helper error → 125 (F3)**
GIVEN `Bin{path, to:None}` and the exec returns `err(1, "--export-path is required")`
WHEN `core::export` runs
THEN `exit_code() == 125` and the tail mentions `export-path`. (Confirms we do NOT invent a
default host dir; we surface the helper's requirement.)

**AC-EXP-12 — usage errors (F5) via the CLI**
GIVEN `assert_cmd` invoking the binary
WHEN `cbox export dev` (no target) → exit 64
AND `cbox export dev --app a --service b` (two targets) → exit 64
AND `cbox export dev --list-apps --to /x` (`--to` w/o `--bin`) → exit 64
AND `cbox export dev --list-apps --delete` (`--delete` w/ list) → exit 64.

**AC-EXP-13 — `--json` shape (mutating + list)**
WHEN `cbox export dev --app firefox --json` runs (mock ok)
THEN stdout is valid JSON with `ok:true, action:"export", mode:"app", target:"firefox", deleted:false, entries:[]`
AND `cbox export dev --list-apps --json` emits `entries` as an array, `target:null`.

**AC-EXP-14 — `--dry-run` prints argv, spawns nothing**
GIVEN global `--dry-run`
WHEN `cbox export dev --app firefox --dry-run` runs
THEN the would-be `distrobox enter … -- distrobox-export --app firefox` argv is printed
AND `MockRunner.call_count()` for the export exec is 0 in DryRun mode (runner returns argv
without recording a spawn) — assert no captured export spawn occurred.

**AC-EXP-15 — bad box name → exit 64 (CLI guard)**
WHEN `cbox export "../evil" --app x` runs
THEN exit 64 with the box-name regex message (before any runner call).

**AC-EXP-16 — backend pin on the invocation**
WHEN `core::export` runs with `backend == Podman`
THEN the recorded call's `env` contains the `DBX_CONTAINER_MANAGER` pin
(`backend.dbx_env()`), matching `core::enter` (`mod.rs:560`).

## 11. Residual risk & open questions

- **OQ-1 (DECISION rulings):** Decisions 1–5 go to the maintainer BEFORE execution. The
  recommended cut is imperative-only (D1), the five-mode surface (D2), box→host (D3), the
  §2.4 exit map (D4), helper-level idempotency (D5). The only ruling that changes the
  file-touch map materially is D1→declarative (adds a serde struct, validation, a diff
  field, apply wiring, and the §8 fingerprint) — flagged as the one fork that is NOT additive
  to the recommended cut.
- **OQ-2 (F2 vs F3 exit code):** §2.4 maps "distrobox-export missing" → 70 and "app not
  found / helper error" → 125. If the maintainer prefers a single 125 for all exec
  non-zero, it is a one-line change (drop the F2 stderr-substring branch). Ruling optional;
  default is the 70/125 split.
- **RR-1 (list-mode parsing robustness):** `distrobox-export --list-apps/--list-binaries`
  output format may carry decoration (headers, paths) across distrobox versions. v6.0 parses
  conservatively (split lines, trim, drop empties) and shows the raw `detail` too, so a
  format skew degrades to "shows the raw helper output" rather than dropping data. Note for
  the implementer to eyeball one real `--list-apps` output during smoke (the `#[ignore]`
  manual smoke test), not in CI.
- **RR-2 (stopped box auto-start latency):** export on a stopped box pays distrobox's
  cold-start cost on the `enter`. Acceptable and unavoidable (same as `enter`/provision).
  No timeout added in v6.0 (the standard `run`, not `run_with_timeout`).
- **RR-3 (orphaned host launchers):** `cbox rm <box>` does NOT clean up host `.desktop`/
  wrapper files the box previously exported (cbox does not own a host-side export registry —
  DECISION-3). Documented honestly in README; flagged as a possible future
  `cbox export prune` (host-side GC). NOT a v6.0 defect — it matches `distrobox-export`'s
  own lifecycle.
- **RR-4 (`--extra-flags`/`--sudo` deferral):** deferred per DECISION-2. The documented
  escape hatch is `cbox enter <box> -- distrobox-export …` for the rare case. v6.1 adds them
  additively (no rework).
