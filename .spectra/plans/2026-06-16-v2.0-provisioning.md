`distrobox enter -- sh -c`, and keeps create-time labels for the create-stamped facts that are
genuinely immutable (image, docker_mode, boxfile_path). This is the load-bearing decision.

### 2.2 In scope (Phase 2 — deep)
- New commands `cbox apply <NAME>` and `cbox up <NAME|--file>`.
- Provision execution engine: `type=shell` and `type=copy`, in Boxfile order, post-create.
- Idempotency: per-step content hash, persisted **guest-side** (DECISION-1), skip-unchanged on re-apply.
- `--force` (re-run all steps) / `--redo <idx>` (re-run one step) / `--no-provision` semantics.
- Boxfile↔live diffing: incremental (packages, provision) vs recreate-required (image, mounts,
  docker-mode, sandbox), with a `--recreate` flow + confirm prompt.
- Partial-failure semantics: stop-on-first-failure (default), partial state recorded so re-apply
  resumes, exit-code + cozy surfacing.
- New `core::` API + `core::provision` module; new typed specs; new `tests/` files.

### 2.3 Sketched / forward-compatible (not deep)
- `type=package` provision steps (the v1.0 schema reserves the enum slot; **not** added in v2.0 —
  packages already flow through create's `--additional-packages`; see §9 R6).
- TUI surfacing of apply/up (P3 — reuses `core::apply` / `core::up` unchanged).

### 2.4 Out of scope / Deferred (v2.0)
- Re-ordering detection of provision steps (insert-in-middle): v2.0 keys idempotency on
  **(index, content-hash)**; an inserted step shifts indices and re-runs the tail. Smarter
  content-addressed ordering is deferred (§9 R3).
- Rollback / undo of a partially-applied box (no transactional guest state in v2.0).
- Parallel provision steps (strictly sequential; provisioning is inherently order-dependent).
- Secrets handling in provision steps (`run` strings are plain; no vault). Deferred.
- Live package *removal* (Boxfile drops a package): v2.0 treats packages as additive-incremental;
  removal requires recreate (§6.3, flagged).

## 3. COMMAND SURFACE (Phase 2 — normative)

Conventions inherited from v1.0 §3.0 unchanged: global `--json`, `-q`, `-v/-vv`, `--no-color`,
`-y`, `--dry-run`, `--backend`. NAME validated client-side (`is_valid_name`, exit 64) before any
spawn. Exit-code taxonomy from `src/error.rs` reused verbatim — **no new exit codes** (§3.4).

### 3.1 `cbox apply <NAME>`

Converge an **existing** box to its Boxfile: diff, then run the incremental parts (packages +
provision). Recreate-class diffs prompt (DECISION-3).

```
cbox apply <NAME> [flags]
cbox apply --file <PATH>          # apply the Boxfile at PATH (its `name` field selects the box)
```

| Flag | Type | Default | Meaning |
|---|---|---|---|
| `NAME` (positional) | string | — (required unless `--file` carries a `name`) | the box to converge |
| `--file <PATH>` | path | resolved from label/XDG | Boxfile source (same resolution as `cbox edit`, reusing `core::resolve_boxfile_path`) |
| `--force` | bool | false | re-run **all** provision steps, ignoring stored hashes (idempotency override) |
| `--redo <IDX>` | int (repeatable) | — | re-run only provision step `IDX` (0-based), ignoring its stored hash |
| `--no-provision` | bool | false | apply package/diff changes but skip the `[[provision]]` list entirely |
| `--recreate` | bool | false | permit recreate-class changes (destroys+recreates the container; requires `-y` or a `[y/N]` confirm) |
| `--dry-run` | bool | false | print the plan (per-step: SKIP/RUN/COPY, plus diff classification) and the would-be argv; spawn nothing |

**Resolution order for the Boxfile:** `--file` → `cbox.boxfile_path` label (via inspect) →
`$XDG_CONFIG_HOME/cbox/boxes/<NAME>/Boxfile.toml`. Reuse `core::resolve_boxfile_path` (already
exists in `src/core/mod.rs`). If none found → exit 65 with a cozy "no Boxfile for this box;
create one with `cbox edit <NAME>`".

**Box-not-found:** if the named box does not exist → exit 69 (`CboxError::box_not_found`) with the
cozy hint "Create it with: `cbox up <NAME>`" (note: `up`, not `create`, since apply implies a
Boxfile exists).

**Human output (success, mixed):**
```
Applying Boxfile for "web-dev" …
  packages   ✓ up to date (git, ripgrep)
  provision  [0] shell  ✓ skipped (unchanged)
  provision  [1] shell  ● ran     rustup default stable           (2.1s)
  provision  [2] copy   ● copied  ./dotfiles/.vimrc → /home/me/.vimrc
✓ Box "web-dev" is up to date (1 step ran, 1 skipped, 1 copied)
```

**`--json` result:** see §13 `apply_json_schema`. Shape:
`{ ok, action:"apply", name, diff:{class, fields:[…]}, steps:[{idx, type, status:"ran|skipped|copied|failed", hash, duration_ms, exit_code?}], recreate_required:bool }`

### 3.2 `cbox up [NAME]`

Create-if-absent **then** apply, in one shot. The "just works" entry point.

```
cbox up <NAME> [create-flags] [apply-flags]
cbox up --file <PATH>             # name/image/etc. all from the Boxfile
```

- Accepts the **union** of `cbox create` flags (v1.0 §3.1) and `cbox apply` flags above.
- **Behavior:** resolve box existence via the list/inspect path. If absent → `core::create`
  (exact P1 path, including label stamping). If present → skip create, go straight to apply
  (an existing box is **not** recreated unless `--recreate`).
- `--dry-run` prints the create plan (if absent) **and** the apply plan, spawning nothing.
- After create, the box is "empty of provisioning" → all provision steps RUN (none skipped),
  populating the guest state file for the first time.

**Existence probe:** reuse `core::list_machine` (filters `label=manager=distrobox`) or a targeted
`inspect`; if `inspect` returns not-found → absent. Prefer `inspect <NAME>` (one spawn) over
listing all.

**Human output (created + provisioned):**
```
✓ Created box "web-dev" (fedora-toolbox:latest, docker: none)
Applying Boxfile for "web-dev" …
  provision  [0] shell  ● ran  rustup default stable  (2.1s)
✓ Box "web-dev" is ready (1 step ran)
  Enter it with:  cbox enter web-dev
```

**`--json` result:** `{ ok, action:"up", created:bool, name, apply:{…apply_json_schema…} }`.

### 3.3 Dry-run semantics (both commands)

`--dry-run` is **plan-only** and must spawn nothing in `Capture` mode that mutates state. It MAY
spawn read-only probes (`inspect`, reading the guest state file) — but DECISION: to keep
`--dry-run` provable under G-NO-NET with zero spawns, the dry-run planner uses the **already-known
inputs** (parsed Boxfile + a single inspect for live labels + a single guest-state read) and then
prints the plan. The per-step provision spawns themselves are replaced by `RunMode::DryRun`
invocations whose `argv` is captured and printed (mirrors the P1 create dry-run path in
`core::create`). A step that would RUN prints its would-be `distrobox enter … -- sh -c "<run>"`
argv; a SKIP prints nothing to spawn.

### 3.4 Exit codes (reuse `src/error.rs::exit` — no new codes)

| Code | Const | When in P2 |
|---|---|---|
| 0 | OK | apply/up succeeded (including all-skipped) |
| 64 | USAGE | invalid NAME; `--json` not meaningful but allowed; `--redo` out of range; `--recreate` without confirm in non-tty without `-y` |
| 65 | DATAERR | Boxfile missing/invalid; recreate-class diff without `--recreate` (DECISION-3); copy `src` missing |
| 69 | UNAVAILABLE | `apply` on a non-existent box |
| 70 | SOFTWARE | distrobox missing (via runner `BinaryNotFound`) |
| 74 | IOERR | guest state-file read/write/parse failure; host `src` read failure |
| 75 | TEMPFAIL | backend unreachable |
| 125 | BACKEND_NONZERO | a provision step (or `<backend> cp`) exited non-zero — **child code surfaced** (§7) |

**Rule (inherited):** a provision step's non-zero exit becomes `CboxError::backend_error(code,
stderr, argv)` → exit 125, never a bare Rust error. The cozy pattern-matcher in
`CboxError::backend_error` is extended with provision-specific signals (§7.3).

## 4. STATE & IDENTITY — where the truth lives (resolves DECISION-1)

### 4.1 The problem the v1.0 sketch missed

v1.0 §8 proposed `cbox.provision.<idx>=<hash>` **container labels** for applied-step hashes.
**This is not implementable:** podman 5.8.2 and docker 29.5.3 both treat container labels as
immutable after create (verified — no `--label` on `update`). You cannot stamp a new label onto a
live, already-provisioned container. The only way to change labels is to recreate the container,
which defeats idempotency (recreate = destroy the very state you are tracking).

### 4.2 Two kinds of truth, two stores

| Truth | Mutability | Store | Written by | Read by |
|---|---|---|---|---|
| **Identity** (image, docker_mode, boxfile_path, cbox version, managed) | Set once at create, immutable | **Container labels** (existing P1 mechanism) | `build_create_argv` (P1, +`cbox.image` added §4.4) | `inspect` / `list` (P1 paths) → the diff's "live" side |
| **Applied-provision state** (per-step hash, last-applied timestamp, last result) | Mutates every apply | **Guest-side state file** | `core::provision` via `enter -- sh -c` write | `core::provision` via `enter -- sh -c` read |

This split is honest: identity is genuinely immutable (it *is* the container), and provision
state is genuinely mutable (it changes every apply). Forcing both into labels was the v1.0 sketch
error.

### 4.3 The guest-side state file

**Path (inside the box):** `${XDG_STATE_HOME:-$HOME/.local/state}/cbox/provision.json`.
Because distrobox shares the host `$HOME` by default, this file lives in the shared home and
persists across `enter` sessions and across host reboots. (Fallback if a box has a custom
`--home` not on a persistent FS: same path under that home; documented sharp edge §9 R2.)

**Schema (`provision.json`):**
```json
{
  "cbox_state_version": 1,
  "boxfile_sha": "sha256:…",          // hash of the whole normalized Boxfile, for fast "any change?" check
  "steps": [
    { "idx": 0, "type": "shell", "hash": "sha256:…", "applied_at": "2026-06-16T…Z", "result": "ok" },
    { "idx": 1, "type": "copy",  "hash": "sha256:…", "applied_at": "…",            "result": "ok" }
  ]
}
```

**Read mechanism:** `distrobox enter --name <N> -- sh -c 'cat "$STATE_PATH" 2>/dev/null || echo ""'`
(Capture). Empty/missing → treat as "no steps applied yet" (first apply). A parse error → IOERR
(74) with a cozy "provision state is corrupt; re-run with --force to reset".

**Write mechanism:** after each step (or once at end — see §7.2 for the failure-resume reason we
write **incrementally**), `distrobox enter --name <N> -- sh -c 'mkdir -p "$DIR"; cat > "$FILE"' `
with the new JSON piped via the command string (the JSON is embedded in the `sh -c` argument,
shell-quoted). Since the runner has no stdin-streaming seam, the write is done by encoding the
JSON as a `printf '%s' '<json>' > "$FILE"` command. **Shell-safety:** the JSON is single-quote
escaped (`'` → `'\''`) before embedding. A new pure helper `build_state_write_argv(name, json)`
lives in `dbox/argv.rs` and is unit-tested for escaping.

> **DECISION-1 fallback (if maintainer rejects guest-home persistence):** a host-side sidecar at
> `$XDG_STATE_HOME/cbox/boxes/<name>/provision.json` is a drop-in alternative — same schema, same
> `core::provision` logic, but read/written with host `std::fs` instead of `enter -- sh -c`. It is
> *simpler to test* (no spawn) but splits state from the box (a box moved/rebuilt loses its host
> sidecar). The spec proceeds on guest-side; switching to host-side touches only the
> `ProvisionStateStore` trait impl (§11.1), not the engine.

### 4.4 New create-time label (additive to P1)

Add one label so image-diff has a labelled source of truth (today image is only readable from
inspect, which is fine, but a label makes the diff uniform):

`--label cbox.image=<image>` appended in `build_create_argv` alongside the existing four labels.
**This is the only change to a P1 file**, and it is purely additive (a new label; no behavior
change to create). All P1 create tests stay green; one new assertion is added.

## 5. PROVISION EXECUTION ENGINE

### 5.1 Execution model

`core::provision(plan, runner)` walks the `[[provision]]` list **in Boxfile order**. For each
step it computes the content hash (§5.2), compares to the stored hash for that index (§4.3), and:

- **hash matches stored** AND not `--force`/`--redo idx` → **SKIP** (record `skipped`).
- **otherwise** → **RUN** (shell) or **COPY** (copy), then on success update the guest state file
  entry for that index (§7.2 incremental write).

The engine is a pure-ish orchestrator: every spawn goes through `&dyn DistroboxRunner` (the P1
seam), so the entire walk is `MockRunner`-provable with zero real distrobox.

### 5.2 Hashing (idempotency key)

Per-step hash = `sha256` over a normalized canonical byte string:

| Step type | Canonical input |
|---|---|
| `shell` | `"shell\n" + normalize(run)` where `normalize` = trim trailing whitespace per line + ensure single trailing `\n`. (Leading whitespace preserved — it can be semantically meaningful in heredocs.) |
| `copy` | `"copy\n" + sha256(host_file_bytes(src)) + "\n" + dst` — hashing **src file content** means editing the copied file re-triggers the step; `dst` change also re-triggers. |

**Hash stability:** the hash is over Boxfile-declared content only (+ copy source bytes), never
over guest runtime output, distrobox version, or timestamps → stable across distrobox/distro drift
(addresses §9 R4). `sha256` is computed with a tiny dependency-free implementation OR the `sha2`
crate (see §11.4 dep decision). The hash is **not** keyed on the box image — changing the image is
a recreate-class diff (§6), handled separately.

**Why `(idx, hash)` not just `hash`:** keying on index keeps the scheme dead-simple and mock-
testable. The cost is that inserting a step in the middle shifts indices and re-runs the tail
(§2.4, R3). Acceptable for v2.0; shell provision steps are expected to be idempotent anyway
(that's the user's contract, like a Vagrant `shell` provisioner).

### 5.3 `type=shell` execution

```
distrobox enter --name <NAME> -- sh -c "<run>"
```
Built by a new pure helper `build_provision_shell_argv(name, run) -> Vec<String>` in
`dbox/argv.rs`. The `run` string is passed as a **single** `sh -c` argument (no host-side shell
splitting — the guest shell parses it), so multi-line `run` and shell operators work. Spawned in
`RunMode::Capture`; `CmdOutput.status` is the step's exit code. Non-zero → stop + exit 125 (§7).

This reuses the **exact** `enter --name <N> -- <cmd>` shape the P1 `build_enter_argv` already
produces and that `MockRunner` already matches on — so no new runner capability is needed.

### 5.4 `type=copy` execution (resolves DECISION-2)

```
<backend> cp <host-src> <NAME>:<guest-dst>
```
Built by `build_copy_argv(backend, src, dst, name) -> Vec<String>`, spawned in `Capture`.

**Why `<backend> cp` and not `enter -- cp`:** `podman cp` and `docker cp` are **identical** in
surface, need **no guest tooling**, work on **stopped** containers, and copy from an arbitrary
**host** path (not requiring the path to be under a mount). The rejected `distrobox enter -- cp`
alternative only works if `src` is already visible inside the box (i.e. under a mount), which is a
fragile precondition. The rejected tar-pipe needs a stdin-streaming seam the runner does not have.

**Pre-flight:** before spawning, `core::provision` checks `src` exists and is readable on the host
(`std::fs::metadata` — runtime I/O, not a codebase mutation, explicitly allowed). Missing src →
exit 65 (DATAERR) with cozy "copy step [i]: source '<src>' not found on host". A non-absolute
`src` is resolved relative to the **Boxfile's directory** (so Boxfiles are portable), `dst` must
be absolute (validated; reuse the mount abs-path rule style).

**Backend resolution for copy:** the container name resolves to the backend chosen by
`Backend::detect` (P1 §4.5). `distrobox`-created containers are visible to the backend it used;
cbox passes `DBX_CONTAINER_MANAGER` through so they stay aligned (P1 §4.5).

### 5.5 No `type=package` in v2.0

The schema enum reserves `package` (v1.0 §6 future), but v2.0 does **not** add it: declared
`packages` already install at **create** time via `--additional-packages`. Adding packages to an
existing box is the **incremental package diff** (§6.3), executed as a synthesized shell step
(distro-aware install command) — but that is keyed off the `packages` field, not a `provision`
entry. Keeping `provision[].type` to `{shell, copy}` matches the P1 `ProvisionType` enum exactly
(no model change). See §9 R6.

## 6. BOXFILE ↔ LIVE DIFFING

### 6.1 The two sides of the diff

| | "Declared" side (Boxfile) | "Live" side (the box) |
|---|---|---|
| Source | parsed `Boxfile` (P1 `boxfile::parse_file`) | container **labels** + **inspect** (P1 `core::inspect` / `project_inspect_json`) + guest **state file** (§4.3) |
| image | `bf.image` | `cbox.image` label (§4.4) → fallback inspect `Config.Image` |
| docker_mode | `bf.docker` | `cbox.docker_mode` label |
| mounts | `bf.mounts` | inspect `Mounts[]` (P1 `parse_mounts`) |
| sandbox/unshare/init | `bf.sandbox` | not fully recoverable from labels (see R5) → treated conservatively |
| packages | `bf.packages` | `cbox.packages` label if present (P1 inspect reads it) |
| provision | `bf.provision` (+ per-step hash) | guest state file `steps[]` |

### 6.2 Diff classification (the core decision)

Each field is classed **INCREMENTAL** (can converge a live box without destroying it) or
**RECREATE** (only achievable by destroy + recreate):

| Field | Class | Why | How compared |
|---|---|---|---|
| `image` | **RECREATE** | a container's image is fixed at create; you cannot swap it live | `bf.image` vs `cbox.image` label (string eq, after registry-prefix normalization) |
| `docker` (mode) | **RECREATE** | socket mounts / `--init` / packages are create-time `distrobox create` flags | `bf.docker.as_str()` vs `cbox.docker_mode` label |
| `mounts` | **RECREATE** | `--volume` is create-time; cannot add a bind mount to a running container | set-compare `bf.mounts` vs inspect `Mounts` (host:guest:mode tuples) |
| `sandbox.unshare` / `sandbox.init` | **RECREATE** | `--unshare-*` / `--init` are create-time namespace flags | best-effort; if not recoverable, **assume unchanged** (R5) — never trigger a surprise recreate on an unrecoverable field |
| `box.home` / `box.hostname` | **RECREATE** | create-time | inspect-derived |
| `packages` | **INCREMENTAL** | can `dnf/apt install` into a live box | set-diff: added packages → install; removed → **RECREATE** (v2.0 limitation §2.4) |
| `provision` | **INCREMENTAL** | the whole point of the engine | per-step hash vs guest state |

### 6.3 Incremental package convergence

If `bf.packages` has **additions** vs the `cbox.packages` label, the engine synthesizes a shell
provision-equivalent install, run **before** the `[[provision]]` list (packages first, then custom
provisioning — matches create ordering):

```
distrobox enter --name <N> -- sh -c "<pkg-install-cmd> <added pkgs…>"
```
`<pkg-install-cmd>` is distro-detected at apply time via a guest probe
(`enter -- sh -c 'command -v dnf || command -v apt-get || command -v apk …'`) → maps to
`sudo dnf install -y` / `sudo apt-get install -y` / `sudo apk add` etc. The probe result is part
of the apply plan; in `--dry-run` it prints the would-be install argv. **Package *removals* are
RECREATE-class in v2.0** (no reliable cross-distro uninstall-to-match-declared semantics) — flagged
to the user, requires `--recreate`.

> **Note:** the package label `cbox.packages` is read by P1 `project_inspect_json` already. v2.0
> does **not** rewrite it on incremental install (labels immutable) — so after an incremental
> package add, the label is stale vs reality. The guest state file records the *effective* package
> set under a new `packages_applied` field so the next diff is accurate. (This is the same
> immutability constraint as DECISION-1, handled the same way.)

### 6.4 Recreate flow (resolves DECISION-3)

When the diff contains any RECREATE-class change:

- **No `--recreate`:** **exit 65** with a cozy explanation listing exactly which fields force a
  recreate, e.g.
  ```
  ✗ "web-dev" needs a recreate to apply these changes:
      image:  fedora-toolbox:latest  →  fedora-toolbox:40
      mounts: + /data:/data:rw
    A recreate destroys the container (your $HOME is preserved; box-local /usr changes are lost).
    Re-run with:  cbox apply web-dev --recreate
  ```
- **With `--recreate`:** prompt `Recreate "web-dev"? This destroys the container. [y/N]` (skip
  prompt if `-y`). On confirm: `core::rm` (force) → `core::create` (from Boxfile-derived
  `CreateSpec`, the exact P1 path) → then run the **full** provision list (fresh box → all steps
  RUN). On decline: exit 0, "left unchanged".
- **Data-loss honesty:** the message always states that `$HOME` is preserved (shared mount) but
  box-local filesystem changes outside `$HOME` are lost. This is the R7 mitigation.

`cbox up --recreate` behaves the same; `cbox up` on an **absent** box never needs recreate (it
creates fresh).

## 7. FAILURE SEMANTICS

### 7.1 Stop vs continue

**Default: stop on first failure.** Provisioning is order-dependent (step N+1 often assumes step N
succeeded), so continuing past a failure produces a misleading "mostly worked" state. A failed
step:
1. stops the walk (later steps are **not** attempted),
2. is recorded in the guest state file with `"result":"failed"` (and the prior successful steps
   keep their `"ok"` entries — see §7.2),
3. surfaces as **exit 125** with the child's exit code + last 5 stderr lines (P1
   `CboxError::backend_error`), prefixed cozily (§7.3).

(No `--continue-on-error` in v2.0 — deferred. A user who wants it splits steps across applies.)

### 7.2 Incremental state write = resume-ability

The guest state file is updated **after each successful step**, not once at the end. So if step 1
of 3 succeeds and step 2 fails:
- the state file records step 0 and step 1 as `ok` (hashes stored),
- step 2 is `failed` (or simply absent/last-attempted),
- a **re-apply** SKIPs steps 0–1 (hashes match) and **resumes at step 2** — exactly the
  Vagrant-style resume the maintainer asked for.

The write itself is a spawn (`enter -- sh -c 'printf … > state'`); if **that** write fails, the
step's work is done but unrecorded → next apply re-runs it. Re-running an already-succeeded
idempotent shell step is safe by the provisioning contract (steps must be idempotent). This is the
honest trade-off of a guest-side store without transactions (§9 R2).

### 7.3 Cozy failure surfacing

Extend `CboxError::backend_error`'s pattern-matcher (in `src/error.rs`) with provision signals
(additive — existing P1 patterns untouched):

| Signal in step stderr | Cozy headline |
|---|---|
| `Permission denied` / `sudo: a password is required` | `Step [i] needs sudo. distrobox usually grants passwordless sudo — check the box's user, or run: cbox enter <N> -- sudo true` |
| `command not found` (in a shell step) | `Step [i]: "<cmd>" not found in the box. Add it to packages, or install it in an earlier step.` |
| copy `no such file` / `not a directory` (guest dst) | `Copy step [i]: destination dir for "<dst>" doesn't exist in the box. Create it in an earlier shell step.` |
| anything else | generic, **always includes the exact `enter … -- sh -c "<run>"` argv** so the user can reproduce by hand |

Human output on failure:
```
Applying Boxfile for "web-dev" …
  provision  [0] shell  ● ran      rustup default stable        (2.1s)
  provision  [1] shell  ✗ failed   cargo install just            (exit 101)
      cargo: error: could not compile `just`
✗ Provisioning stopped at step [1] of 3. Fix the Boxfile and re-run:  cbox apply web-dev
  (steps 0 are recorded; re-apply resumes at step 1)
```

### 8.4 Dependency decision (no heavy deps)

| Need | Decision |
|---|---|
| sha256 | **Recommend `sha2` crate** (tiny, no transitive weight, ubiquitous, audited) over a hand-rolled sha256. This is the one new runtime dep. **Justification:** correctness of a hash that gates idempotency must not rest on a bespoke implementation. If the maintainer wants zero new deps, a vendored ~100-line sha256 is the fallback (flagged). |
| JSON state file | reuse existing `serde_json` (already a P1 dep). |
| timestamps | `applied_at` uses a simple RFC3339 string from… **no chrono.** Use a minimal format via `std::time::SystemTime` → seconds-since-epoch string, or omit human timestamps and store epoch secs. Avoids a `chrono`/`time` dep. |

**Deliberately excluded (unchanged from P1):** no async/tokio, no network crates, no config
framework. The guest state file + labels remain the only state stores.

## 10. ACCEPTANCE CRITERIA (GIVEN/WHEN/THEN — mock-testable)

All against `MockRunner`; zero real distrobox (G-NO-NET). The mock's queued-matcher + recorded-
calls API (`src/dbox/mock.rs`) drives multi-step walks; the state store is mocked via a test impl
of `ProvisionStateStore` OR via mock-intercepted `enter -- sh -c cat/printf` spawns.

### apply → `tests/apply.rs`
- **AC-APPLY-1** GIVEN an existing box and a Boxfile with 2 shell steps, AND a state store reporting **no** applied hashes, WHEN `cbox apply web-dev` THEN the runner is called with two `distrobox enter --name web-dev -- sh -c "<run>"` invocations in order, exit 0, and "ran" status for both.
- **AC-APPLY-2** GIVEN the same Boxfile AND a state store whose stored hashes **match both** steps, WHEN `cbox apply web-dev` THEN **no** `enter -- sh -c "<run>"` provision spawn occurs (only the state read), exit 0, both "skipped". (idempotency / G-IDEMPOTENT)
- **AC-APPLY-3** GIVEN matching hashes AND `--force`, WHEN apply runs THEN both steps RUN regardless of stored hashes.
- **AC-APPLY-4** GIVEN a 3-step Boxfile AND `--redo 1`, WHEN apply runs THEN only step 1 RUNs; steps 0 and 2 SKIP (their hashes match).
- **AC-APPLY-5** GIVEN a Boxfile whose `image` differs from the `cbox.image` label AND no `--recreate`, WHEN apply runs THEN exit 65 with a cozy recreate-required message naming `image`, and **no** provision spawn occurs. (DECISION-3 / G-EXIT)
- **AC-APPLY-6** GIVEN a recreate-class diff AND `--recreate -y`, WHEN apply runs THEN the runner sees `distrobox rm --force web-dev` then `distrobox create … web-dev` then the full provision list. 
- **AC-APPLY-7** GIVEN `apply` on a box the runner reports as not-found, WHEN apply runs THEN exit 69 with the "create it with: cbox up" hint, no provision spawn.
- **AC-APPLY-8** GIVEN `--no-provision` AND an added package in the Boxfile, WHEN apply runs THEN the package-install spawn occurs but **no** `[[provision]]` step spawn occurs.
- **AC-APPLY-9** GIVEN `--dry-run`, WHEN apply runs THEN every provision spawn is `RunMode::DryRun` (argv captured, nothing executed) and the plan is printed; assert no `Capture`-mode `enter -- sh -c "<run>"` mutation occurred.
- **AC-APPLY-10** GIVEN `--json`, WHEN apply runs THEN stdout is one JSON doc matching `apply_json_schema` (steps[] with idx/type/status/hash), chrome on stderr only.

### up → `tests/up.rs`
- **AC-UP-1** GIVEN the runner reports the box **absent**, WHEN `cbox up web-dev` THEN `distrobox create … web-dev` is called first, then the full provision list runs (all "ran"), exit 0, output includes both "Created" and "ready".
- **AC-UP-2** GIVEN the runner reports the box **present**, WHEN `cbox up web-dev` THEN **no** `create` spawn occurs; behavior is identical to `apply`.
- **AC-UP-3** GIVEN `cbox up --file Boxfile.toml` where the Boxfile carries the name, WHEN run THEN name/image/etc. derive from the Boxfile (reuse `spec_from_boxfile`) and create+provision proceed.
- **AC-UP-4** GIVEN `cbox up web-dev --dry-run` on an absent box THEN both the create plan and the provision plan print, zero mutating spawns.

### provision engine → `tests/provision.rs`
- **AC-PROV-1** (hashing) `hash_step(shell{run})` is a pure deterministic function; identical `run` → identical hash; whitespace-normalized `run` variants → identical hash; different `run` → different hash.
- **AC-PROV-2** (copy hash) `hash_step(copy{src,dst})` changes when the **src file content** changes (write two temp files with different bytes) and when `dst` changes; stable when neither changes.
- **AC-PROV-3** (copy spawn) GIVEN a copy step with an existing host `src`, WHEN the engine runs THEN the runner is called with `<backend> cp <src> web-dev:<dst>`.
- **AC-PROV-4** (copy missing src) GIVEN a copy step whose host `src` does not exist, WHEN the engine runs THEN exit 65 (DATAERR) before any spawn.
- **AC-PROV-5** (partial-failure resume / G-RESUME) GIVEN a 3-step walk where the mock makes step 1 exit non-zero, WHEN apply runs THEN exit 125, step 0 RAN, step 2 was **not** attempted, AND the state-store write recorded step 0 as ok; THEN a 2nd apply with that partial state SKIPs step 0 and resumes at step 1.
- **AC-PROV-6** (state read) GIVEN the mock returns a `provision.json` with one stored hash, WHEN the engine reads state THEN it parses and step 0 is considered already-applied.
- **AC-PROV-7** (state write escaping / R8) `build_state_write_argv(name, json)` single-quote-escapes the JSON; a JSON containing a `'` produces a shell-safe `sh -c` argument (golden assertion).
- **AC-PROV-8** (corrupt state) GIVEN the mock returns non-JSON for the state read, WHEN the engine runs THEN exit 74 with a cozy "corrupt; --force to reset", unless `--force` (then treated as empty).

### diff → `tests/diff.rs`
- **AC-DIFF-1** GIVEN equal Boxfile and live (same image label, same mounts), WHEN diffed THEN `Diff.class == Incremental` and `fields` is empty.
- **AC-DIFF-2** GIVEN a differing `image`, WHEN diffed THEN `class == Recreate` and `fields` contains an `image` entry with old/new.
- **AC-DIFF-3** GIVEN a differing `docker` mode, WHEN diffed THEN `class == Recreate` naming `docker`.
- **AC-DIFF-4** GIVEN an added mount, WHEN diffed THEN `class == Recreate` naming `mounts`.
- **AC-DIFF-5** GIVEN only added `packages`, WHEN diffed THEN `class == Incremental` and a `packages` field lists the additions.
- **AC-DIFF-6** GIVEN a *removed* package, WHEN diffed THEN `class == Recreate` (v2.0 limitation §2.4) naming `packages` removal.
- **AC-DIFF-7** GIVEN an unrecoverable `sandbox.unshare` live value, WHEN diffed THEN it is **assumed unchanged** (no recreate triggered) (R5).
