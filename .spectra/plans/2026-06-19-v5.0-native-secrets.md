**Spec ID:** `2026-06-19-v5.0-native-secrets`
**Status:** DECISION-READY · AUTO_PROCEED (confidence 0.88)
**Intent:** `CHANGE` (extend existing crate) · Complexity 9/12 (extended-depth)
**Author:** SPECTRA · **Executor:** Vivi (Builder/Reasoner)
**Build:** containerized only — `make test` / `make check`, never bare `cargo` on host.

> All product/architecture decisions (D0–D4) are LOCKED upstream. This spec maps
> them onto concrete file-level contracts. The injection mechanism is EMPIRICALLY
> VERIFIED (podman 5.8.2 + distrobox 1.8.2.4): the secret VALUE rides the
> cbox→distrobox child-process environment; only the KEY NAME appears in argv.

## 0. Scope

### In scope (v1)
- `[secrets]` Boxfile table (KEY-only references) + optional `[env]` plaintext table.
- A bundled, pure-Rust Secret Service client behind an internal `SecretStore` trait.
- `cbox secret set | list | rm` command surface.
- Injection wiring into the `create` path (persist=true → `Config.Env`) and the
  `provision enter` path (persist=false → exec-session only).
- A value-free secrets/env fingerprint that participates in the diff/converge decision.
- Doctor Secret Service availability line.
- Keyring-unavailable refusal (exit 75) and the structural no-value-in-argv invariant.

### Out of scope / Deferred (D4 — lean v1)
- Prompt-as-a-source (`from = "prompt"`). Reserved in the schema, rejected by validation.
- Value rotation as a diff concern (explicit user workflow instead — §7).
- Per-secret TTL, secret sharing across boxes, secret templating/interpolation.
- Masking secret values inside arbitrary `[[provision]]` `run` strings the user authors.
- Multiple shipped keyring backends. The `SecretStore` trait is a TEST SEAM only;
  exactly one real impl ships (the crate). (D1)

### Assumptions (risk-if-wrong)
- **A1** The host runs a freedesktop Secret Service provider (gnome-keyring / KWallet
  via Secret Service / KeePassXC). Risk-if-wrong: every secret op refuses with exit 75.
  This is the intended, documented behavior (D3), not a defect.
- **A2** The verified `--env KEY` (name-only) child-env mechanism holds on the
  podman/docker + distrobox versions cbox supports (floor 1.6). Risk-if-wrong: a value
  could land in argv. Mitigated by the hard no-value-in-argv test invariant (§9, INV-1)
  which fails the build if any future change regresses it.

## 1. Boxfile schema additions

### 1.1 Narrative

Two new optional top-level tables. Both default to empty.

```toml
name  = "api-dev"
image = "registry.fedoraproject.org/fedora-toolbox:latest"

# Secret KEY references. Values live ONLY in the OS keyring (D0).
[secrets]
DATABASE_URL = { persist = true }                  # persist defaults to true
STRIPE_KEY   = { persist = false }                 # provision-time only
GITHUB_TOKEN = { persist = true, from = "keyring" } # `from` is explicit but optional

# Plaintext, NON-secret env. Values are fine in argv (not secret).
[env]
EDITOR = "nvim"
LANG   = "en_US.UTF-8"
```

- **`[secrets]`** — each entry is an inline table `{ persist = <bool>, from = "keyring" }`.
  - `persist` — optional bool. **DEFAULT = `true`** (DECISION below).
  - `from` — optional string, defaults `"keyring"`, the only legal value in v1.
    `"prompt"` and any other value are a data-error (exit 65), so the field is
    forward-reserved without being silently accepted.
  - The **VALUE never appears** in the Boxfile. Only the KEY (the table key) and the
    two flags. This is the D0 invariant made structural by the schema shape.
- **`[env]`** — each entry is a plain `KEY = "value"` string. The value DOES appear in
  the committed file and in argv (`--env KEY=value`). Non-secret by definition.

#### DECISION — `persist` default = `true`
Rationale: matches the user's stated "set and forget" reliability goal. A
`persist=true` secret is present in every `cbox enter` and every provision step with no
further action, which is the least-surprising behavior for the common case (a DB URL, a
registry token). `persist=false` is the deliberate opt-in for the stricter
"provision-time only, never in the persistent container env" posture. The cost of the
default: a `persist=true` secret is baked into `Config.Env` and therefore visible to
`podman inspect` on the host (§6.ii) — this is inherent to availability and is documented,
not hidden.

#### DECISION — include `[env]` in v1 (do NOT defer)
Rationale: (a) it is nearly free — it reuses the exact same `--env KEY=value` create-path
plumbing as secrets minus the keyring round-trip, and the value-in-argv is acceptable
because it is non-secret; (b) it closes the obvious "but where do I put `EDITOR`?" gap
that would otherwise push users to abuse `[secrets]` for non-secrets (forcing a keyring
write for a trivial value); (c) it lets the duplicate-key validation rule (§2) be
meaningful — `[env]` and `[secrets]` share a namespace, so a coherent v1 must define
both to police collisions. The added surface is one serde map + one validation loop + one
create-path branch.

### 1.2 Serde representation

New file `src/boxfile/model.rs` additions:

```rust
// top-level Boxfile additions
#[serde(default)]
pub secrets: std::collections::BTreeMap<String, SecretEntry>,
#[serde(default)]
pub env: std::collections::BTreeMap<String, String>,

/// A [secrets] entry — KEY-only reference; value lives in the keyring (D0).
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
#[serde(deny_unknown_fields)]                 // typo guard, per sub-table convention
pub struct SecretEntry {
    #[serde(default = "default_persist")]     // DEFAULT = true
    pub persist: bool,
    #[serde(default = "default_from")]        // "keyring"
    pub from: String,
}
fn default_persist() -> bool { true }
fn default_from() -> String { "keyring".to_string() }
```

- **Use `BTreeMap`** (not `HashMap`): deterministic key ordering is required so the
  fingerprint (§7), the create-path `--env KEY` flag order, and test assertions are
  stable across runs. (TOML preserves no order; BTreeMap gives a canonical one.)
- **`SecretEntry` gets `#[serde(deny_unknown_fields)]`** — matches the existing
  sandbox/box sub-table convention (`model.rs:94,149`) and catches `persit = true` typos
  as a hard error instead of a silent default.
- **`[env]`** is `BTreeMap<String, String>` — no wrapper struct; a bare string value.

### 1.3 Top-level `deny_unknown_fields` — DECISION: keep OFF; extend the allowlist

The top-level `Boxfile` struct intentionally does NOT `deny_unknown_fields` (forward-compat
warnings, `model.rs:1`). Do **not** flip it on — that is an orthogonal breaking change and
out of scope. Instead, add `"secrets"` and `"env"` to the `known_top_level` allowlist in
`boxfile/mod.rs:15-24` so they are recognized (no spurious "unknown key" warning). Unknown
keys keep warning as today.

## 2. Validation rules (extend `src/boxfile/validate.rs`)

All failures are `CboxError::dataerr` → **exit 65**, joined into the existing `errors`
vector so multiple problems surface at once (matches current `validate()` shape).

Add a constant: `const ENV_NAME_REGEX: &str = "^[A-Za-z_][A-Za-z0-9_]*$";` and a helper
`fn is_valid_env_name(k: &str) -> bool` (first char alpha/underscore, rest alnum/underscore).

| # | Rule | Error text (exit 65) |
|---|------|----------------------|
| V1 | Every `[secrets]` KEY and `[env]` KEY matches `^[A-Za-z_][A-Za-z0-9_]*$` | `secrets."<KEY>": invalid env-var name. Must match ^[A-Za-z_][A-Za-z0-9_]*$` (and the `env."<KEY>"` variant) |
| V2 | No KEY appears in BOTH `[env]` and `[secrets]` | `"<KEY>" is declared in both [env] and [secrets]; a key may be one or the other, not both.` |
| V3 | `secrets.<KEY>.from` ∈ {`"keyring"`} | `secrets."<KEY>".from = "<v>" is not supported in this version. Only "keyring" is valid (prompt-as-source is deferred).` |
| V4 | Count bound: `secrets.len() + env.len() <= 64` | `Too many env/secret entries (<n>). The limit is 64.` |
| V5 | KEY length bound: `1 <= KEY.len() <= 128` | `env/secret key "<KEY>" is too long (<n> chars). Max 128.` |
| V6 | `[env]` value length bound: `value.len() <= 4096` | `env."<KEY>" value is too long (<n> bytes). Max 4096.` |

Notes:
- V1 uses the **stricter POSIX env-var regex** (no leading digit, no `.`/`-`), which is
  deliberately tighter than the box-NAME regex — these become shell-exported variable
  names inside the guest, so they must be legal identifiers.
- V2 iterates `secrets.keys()` and checks `env.contains_key`. BTreeMap makes the
  collision report order-stable.
- The bounds (V4/V5/V6) are sanity guards against pathological Boxfiles, not security
  boundaries. They are cheap and prevent a 10,000-entry argv blowup.
- **No keyring access happens during validation.** Validation is pure (matches the rest
  of `validate.rs`). Whether a referenced secret actually EXISTS in the keyring is checked
  at injection time (§3/§5), not here — so `cbox apply --dry-run` and validation work
  offline.

## 3. Keyring storage contract

### 3.1 Crate selection — DECISION

**Use the `keyring` crate v3, with default features OFF and these features ON:**

```toml
# Cargo.toml [dependencies]
keyring = { version = "3", default-features = false, features = [
    "sync-secret-service",   # Linux Secret Service over zbus, blocking API
    "crypto-rust",           # pure-Rust AES via RustCrypto (encrypted transport)
] }
```

**Justification (static / no-C-link, D1):**
- `sync-secret-service` selects the `secret-service` crate which talks to the Secret
  Service D-Bus API via **`zbus` (pure Rust)** — there is **no link against libsecret**
  and **no `secret-tool` runtime prerequisite**. The freedesktop Secret Service is reached
  over the session D-Bus socket directly.
- `crypto-rust` selects **RustCrypto** for the encrypted-session crypto, so there is **no
  OpenSSL / no C crypto link**. (The alternative `crypto-openssl` is explicitly NOT
  selected.)
- `default-features = false` drops the platform backends we do not ship (macOS keychain,
  Windows credential store, the `linux-native`/libsecret path). The result keeps the cbox
  binary a single self-contained artifact — consistent with the existing
  `strip = true` release profile and ARM cross-builds (`Cargo.toml:38`).
- This is preferred over depending on `secret-service` directly because `keyring`'s
  `Entry` API gives us the cross-process credential abstraction for free and a single,
  well-maintained dependency surface; we still get the pure-Rust transport because the
  feature flags select it transitively.

> RESIDUAL: pin the minor version after Vivi confirms `make build` resolves the lockfile
> inside the cbox-dev image (the crate must vendor cleanly into the offline container
> build). See §11 OQ-1.

### 3.2 Namespacing — DECISION

A secret is keyed by **service = `"cbox"`** and a **user/account string of
`"<box>/<KEY>"`**:

```rust
keyring::Entry::new("cbox", &format!("{box_name}/{key}"))
```

- Service `"cbox"` groups all cbox-managed secrets under one Secret Service collection
  label, so a user can audit them in `seahorse`/KWallet by service.
- Encoding both `box` and `KEY` in the account string namespaces per-box (two boxes may
  each have a `DATABASE_URL` without collision) while keeping a single, greppable key
  shape. `/` is chosen as the separator because it cannot appear in a valid box NAME
  (`^[a-zA-Z0-9][a-zA-Z0-9_.-]*$`, no slash) nor in a valid env KEY (V1 regex, no slash),
  so `"<box>/<KEY>"` is unambiguously parseable back into `(box, KEY)` — required for
  `list(box)` (§3.3).
- `list(box)` is implemented by enumerating the collection and filtering account strings
  with prefix `"<box>/"` (see §3.3 note on the `keyring` search API).

### 3.3 The `SecretStore` trait (TEST SEAM — D1)

New file `src/secret/mod.rs` (or `src/core/secret.rs`; spec assumes `src/secret/`).

```rust
/// Test seam over the OS keyring. Exactly ONE real impl ships (KeyringStore);
/// the mock exists for tests only (D1 — NOT a multi-backend abstraction).
pub trait SecretStore: Send + Sync {
    fn set(&self, box_name: &str, key: &str, value: &str) -> Result<(), SecretError>;
    fn get(&self, box_name: &str, key: &str) -> Result<Option<String>, SecretError>;
    fn delete(&self, box_name: &str, key: &str) -> Result<(), SecretError>;
    fn list(&self, box_name: &str) -> Result<Vec<String>, SecretError>; // KEY names only
}
```

**Error type → exit codes:**

```rust
#[derive(Debug, thiserror::Error)]
pub enum SecretError {
    /// Secret Service unreachable / locked / no provider (D3).
    #[error("{0}")]
    Unavailable(String),                 // -> exit 75 (TEMPFAIL)
    /// A referenced KEY has no stored value.
    #[error("{0}")]
    NotFound { box_name: String, key: String }, // -> exit 75 at INJECTION (D3 below)
    /// Any other backend failure.
    #[error("keyring backend error: {0}")]
    Backend(String),                     // -> exit 70 (SOFTWARE)
}
```

These map into `CboxError` at the call boundary:
- `Unavailable` and `NotFound` (during injection) → `CboxError::tempfail(...)` → **exit 75**.
  This satisfies D3: the box is NOT created/entered, NOTHING runs, and the message is
  actionable. The `keyring` crate distinguishes `Error::NoEntry` (→ `NotFound`) from
  D-Bus/platform failures (→ `Unavailable`).
- `Backend` → `CboxError::software(...)` → exit 70.

**D3 injection-time refusal message** (the exact contract — names the KEY and the fix):

```
Secret "DATABASE_URL" for box "api-dev" is not available in the keyring.
Unlock your keyring, or store it with:  cbox secret set api-dev DATABASE_URL
(cbox never falls back to a plaintext value — nothing was created or run.)
```

For a locked/missing Secret Service (`Unavailable`):

```
Can't reach the OS keyring (Secret Service). Unlock it (log into your desktop
session / start gnome-keyring) and retry. Secret "DATABASE_URL" for "api-dev"
could not be read.  cbox doctor   shows keyring status.
```

Both are exit 75. The injection caller resolves secrets EAGERLY and ALL-OR-NOTHING
**before** any `distrobox create`/`enter` spawn, so a missing secret aborts with zero
side effects (D3 "run NOTHING").

### 3.4 Real impl + mock

- `KeyringStore` — wraps `keyring::Entry`. `set` → `entry.set_password(value)`;
  `get` → `entry.get_password()` mapping `Error::NoEntry` to `Ok(None)`; `delete` →
  `entry.delete_credential()` (idempotent: NoEntry → Ok); `list` → collection search by
  service `"cbox"` filtered on account-prefix `"<box>/"`, returning the suffixes.
- `MockSecretStore` — an in-memory `Mutex<BTreeMap<(String,String), String>>` plus a
  `fail_mode: Option<SecretError>` switch so tests can force `Unavailable`/`NotFound`.
  Lives under `#[cfg(any(test, feature = "testkit"))]` mirroring `MockRunner` (`mock.rs:4`).

> RESIDUAL OQ-2: `keyring` v3's collection-search surface for `list` differs slightly

## 4. `cbox secret` command surface (lean)

New subcommand `Secret(secret::SecretArgs)` registered in `Commands` (`cli/mod.rs:110`),
dispatched in `main.rs` (`dispatch`, `main.rs:59`). New file `src/cli/secret.rs`.

### DECISION — ship `set` + `list` + `rm` (minimal coherent CRUD)

```
cbox secret set  <BOX> <KEY>     # read value from hidden TTY prompt OR stdin (piped)
cbox secret list <BOX>           # KEY names only — NEVER values
cbox secret rm   <BOX> <KEY>     # delete one key from the keyring
```

`update` is just `set` again (overwrite); no separate verb. No `get`/`show` verb exists —
there is deliberately NO command that prints a value (defense in depth).

#### `cbox secret set <BOX> <KEY>`
- **TTY:** prompt `Value for <KEY> (input hidden): ` on stderr, read with **echo
  disabled**, no newline echoed. Use the `rpassword` crate (pure-Rust, no C link) for the
  hidden read. Add to Cargo.toml: `rpassword = "7"`.
- **Non-TTY (piped/CI):** when `stdin` is not a terminal (reuse the `IsTerminal` check
  already imported in `output.rs:3`), read the value from stdin: take everything up to the
  first newline (so `echo -n "$VAL" | cbox secret set box KEY` and
  `printf '%s\n' "$VAL" | ...` both work), trim a single trailing `\n`/`\r\n`. Do NOT
  echo. This is the scripting path.
- Empty value → exit 64 usage error: `Refusing to store an empty value for "<KEY>".`
- Validates `<KEY>` against the V1 env-name regex BEFORE touching the keyring (exit 65 on
  bad name). Validates `<BOX>` against the box-name regex (exit 64).
- Stores via `SecretStore::set`. Success line (human): `Stored secret "<KEY>" for "<BOX>".`
  Note: cbox does NOT know `persist` here — `persist` lives in the Boxfile, not the
  keyring (§3 stores only the value). Output says nothing about persistence.
- `--json`: `{ "ok": true, "action": "secret-set", "box": "<BOX>", "key": "<KEY>" }`.
  The value is NEVER in JSON output.

#### `cbox secret list <BOX>`
- Prints KEY names only, one per line (human) / `{ "ok": true, "box":"<BOX>",
  "keys": ["A","B"] }` (`--json`). **Never values.**
- Because `persist` is a Boxfile property and the keyring stores only values, list shows
  only keys present in the keyring; it does NOT show `persist`. If a Boxfile is
  discoverable (cwd `Boxfile.toml`), list MAY annotate each key as
  `present-in-boxfile: yes/no` (human mode only) as a courtesy — OPTIONAL, low priority,
  no value ever printed. Default: just key names.
- Empty → `No secrets stored for "<BOX>".` (human) / `"keys": []` (json).

#### `cbox secret rm <BOX> <KEY>`
- `SecretStore::delete`. Idempotent: deleting an absent key is success (`Removed secret
  "<KEY>" for "<BOX>" (or it was not set).`), so scripts are re-runnable.
- `--json`: `{ "ok": true, "action": "secret-rm", "box":"<BOX>", "key":"<KEY>" }`.

#### Exit codes (all three)
- 0 success · 64 bad usage/empty value/bad box name · 65 bad KEY name · 75 keyring
  unavailable/locked · 70 other backend error.
- Honor existing global flags: `--json`, `-q/--quiet`, `--no-color`, `-v`. `-v` may echo
  the **command shape but NEVER the value** (there is no runner argv here; nothing to echo
  beyond `secret set <box> <KEY>`).

## 5. Injection wiring (the verified mechanism)

The core principle: **`--env KEY` is name-only in argv; the VALUE rides
`Invocation.env`** via the existing `with_env` seam (`runner.rs:35`, applied
`real.rs:22-24,55-57`). This is the SAME seam already used for `DBX_CONTAINER_MANAGER`.

### 5.0 Resolution helper (shared by both paths)

New `core` function (in `src/core/secret_inject.rs` or folded into `core::secret`):

```rust
/// Resolve secrets eagerly, ALL-OR-NOTHING, before any spawn (D3).
/// Returns the (KEY, VALUE) pairs to attach to Invocation.env and the KEY names
/// to add as `--env KEY` flags. Errors -> exit 75 (keyring unavailable / key missing).
pub fn resolve_secret_env(
    box_name: &str,
    secrets: &BTreeMap<String, SecretEntry>,
    want: SecretScope,          // Persisted | ProvisionOnly
    store: &dyn SecretStore,
) -> Result<Vec<(String, String)>, CboxError>;
```

`SecretScope::Persisted` selects `persist == true`; `ProvisionOnly` selects
`persist == false`. The function calls `store.get` for each selected key; a `None` →
`CboxError::tempfail` (D3 missing-key message, §3.3); a store `Unavailable` →
`CboxError::tempfail` (D3 unavailable message). It NEVER returns a partial set.

### 5.1 (a) Create path — `persist == true` secrets + all `[env]`

Change the create flow so it carries the env to attach. Concretely:

1. **`CreateSpec` gains two fields** (`src/core/spec.rs`):
   ```rust
   pub env_flags: Vec<String>,            // KEY names -> `--env KEY` in argv (name-only)
   pub env_values: Vec<(String, String)>, // KEY=VALUE -> Invocation.env (NOT in argv)
   ```
   - For each `[env]` entry `K = "v"`: push `format!("{K}={v}")` is the **value-in-argv**
     case (non-secret, allowed) → handled in argv as `--env K=v` (a SINGLE additional-flag
     token, value inline). So `[env]` does NOT use `env_values`; it uses an argv flag with
     the value inline.
   - For each `persist=true` secret `K`: push `"K"` to `env_flags` (name-only) AND
     `("K", <resolved value>)` to `env_values`.

   To keep argv builders pure (they take no keyring), the CLI/`core` layer resolves
   secrets and populates these vectors BEFORE building argv. `build_create_argv`
   (`argv.rs:10`) is extended to, after the existing docker-mode `extra_flags` loop
   (`argv.rs:44-47`), emit:
   ```
   for k in &spec.env_flags  { args.push("--additional-flags"); args.push(format!("--env {k}")); }
   for (k,v) in &spec.plain_env { args.push("--additional-flags"); args.push(format!("--env {k}={v}")); }
   ```
   where `plain_env` holds the `[env]` inline pairs (value-in-argv OK) and `env_flags`
   holds secret KEY names (name-only).

2. **`core::create`** (`core/mod.rs:39`) attaches the values to the invocation:
   ```rust
   let inv = Invocation::new("distrobox", args, mode).with_env(env_for_create);
   ```
   where `env_for_create = spec.env_values` (secret VALUES) — distrobox's child sees
   `KEY=value` in its process env, and because the argv carries `--env KEY` (name only),
   podman/runc bakes `KEY=value` into `Config.Env`. **VERIFIED.** `[env]` plaintext values
   need NOT be on `Invocation.env` (they are already inline in `--env K=v`), but attaching
   them too is harmless; spec keeps them argv-inline only for clarity.

3. **Resolution site** — the CLI `create`/`up` handlers (`cli/create.rs`, `cli/up.rs`) and
   the `apply --recreate` path (`core/mod.rs:872`, `create_spec_from_boxfile_and_apply_spec`
   `core/mod.rs:1053`) must, after building the `CreateSpec` from the Boxfile, call
   `resolve_secret_env(name, &bf.secrets, Persisted, store)` and set
   `spec.env_flags` / `spec.env_values`, and set `spec.plain_env` from `bf.env`. If
   resolution errors (exit 75), the whole command aborts BEFORE `core::create` spawns
   anything (D3).

### 5.2 (b) Provision path — `persist == false` secrets

`persist=false` secrets must reach provision-time `distrobox enter` exec sessions but
NEVER `Config.Env`.

1. **`build_provision_shell_argv`** (`argv.rs:186`) is NOT changed in shape, but the
   provision engine adds `--additional-flags "--env KEY"` for each provision-only secret.
   New builder `build_provision_shell_argv_with_env(name, run, env_keys: &[String])`:
   ```
   enter --name <N> --additional-flags "--env K1" --additional-flags "--env K2" -- sh -c "<run>"
   ```
   (distrobox's `enter --help` documents `--additional-flags "--env MY_VAR=value"`; we use
   the name-only form so the value stays out of argv — VERIFIED.)

2. **`ProvisionPlan`** (`provision.rs:20`) gains `provision_env: &'a [(String, String)]`
   (resolved persist=false KEY=VALUE pairs) and `provision_env_keys: &'a [String]` (the
   KEY names for the `--env KEY` flags). `run_step` (`provision.rs:202`) for `Shell` steps:
   - builds argv with `provision_env_keys`,
   - constructs `Invocation::new("distrobox", args, mode).with_env(provision_env_pairs)`
     so the exec session sees the values.
   Copy steps (`provision.rs:247`) need **NO env** — `<backend> cp` moves files only; no
   secret is relevant. They stay exactly as today.

3. **`persist=true` secrets are ALREADY in `Config.Env`** (baked at create, §5.1), so a
   plain `distrobox enter` for a provision step inherits them automatically — the
   provision engine does NOT re-inject persist=true secrets. State this explicitly: the
   provision env attached is ONLY the persist=false set. (If both sets were attached, the
   persist=true ones would merely be redundant, but keeping them separate keeps the
   "persist=false is provision-only" boundary crisp and avoids any persist=true value
   appearing on the provision invocation's env where it is not needed.)

4. **Resolution site** — `core::apply` / `apply_fresh` / `up` resolve
   `resolve_secret_env(name, &bf.secrets, ProvisionOnly, store)` ONCE before the provision
   walk and thread the result into `ProvisionPlan`. Same all-or-nothing exit-75 semantics
   (D3): a missing persist=false secret aborts the apply before the first step runs.

### 5.3 (c) Order / precedence

- Within argv, secret/env `--additional-flags` are appended AFTER docker-mode
  `extra_flags` and BEFORE the cbox label `--additional-flags` block (`argv.rs:97-102`),
  preserving the existing label tail. Order among secrets/env is BTreeMap key order
  (deterministic).
- If a KEY is in both `[env]` and `[secrets]`, validation V2 already rejected it (exit 65)
  — so no runtime precedence question arises.
- `persist=true` secret + same-named `[env]`? Forbidden by V2. No collision possible.

### 5.4 (d) HARD INVARIANT (asserted in tests — §9 INV-1)

> **INV-1 — No secret VALUE ever appears in any recorded argv.** For every create and
> every provision invocation, the recorded `args` (and `program`) contain only `--env KEY`
> (name-only) for secrets — never `KEY=<value>`. The VALUE appears ONLY in the recorded
> `env` vector. `[env]` plaintext values MAY appear in argv (they are non-secret).

This is checkable directly via `MockRunner`'s `RecordedCall { args, env }` (`mock.rs:11`):

## 6. Redaction posture

The structural mechanism (§5) keeps secret values out of argv, which neutralizes the two
argv-based leak sinks WITHOUT a runtime scrubber:

- **(i) S1 (step-failure error `error.rs:117-132`) and S2 (`-v` argv echo
  `output.rs:84-88`) need NO value-scrubber.** Both render argv only; argv has no secret
  values by construction. Instead of a scrubber we assert INV-1 (§5.4, §9). This is
  strictly better than a regex redactor (no false negatives, no perf cost, no value ever
  in a buffer that could be logged).
- **(ii) S3 `inspect --raw` (`core/mod.rs:551-560` / `cli/inspect.rs:33-37`).** A
  `persist=true` secret is inherently in `Config.Env` and therefore visible to
  `podman/docker inspect`. **DECISION: cbox masks KNOWN secret keys in its OWN
  `inspect --raw` output as a courtesy.** "Known" = keys named in `[secrets]` of the box's
  resolved Boxfile (discovered via the `cbox.boxfile_path` label, same lookup as
  `resolve_boxfile_path` `core/mod.rs:739`). For each known secret KEY, rewrite any
  `"KEY=<value>"` element of the JSON `Config.Env` array to `"KEY=***"` before printing.
  - Implementation: parse the raw JSON, walk `Config.Env`, mask, re-serialize. If the
    Boxfile is not discoverable (no label, file gone), mask nothing and print as-is (we
    cannot know which keys are secret) — and emit a one-line stderr note in human mode:
    `note: could not resolve Boxfile; secret values in Config.Env are not masked.`
  - **Document loudly** (README + the masked-output note): `cbox inspect --raw` masking is
    a courtesy on cbox's OWN output; **host `podman inspect <box>` still shows the real
    values** because they genuinely live in `Config.Env`. This is the inherent cost of
    `persist=true` and is by design (availability beats secrecy-from-host-root for the
    persisted set).
- **(iii) Projected `inspect --json` (`core/mod.rs:493-508`, `spec.rs:133-153`) stays
  safe** — `InspectResult` has NO env field; it only extracts `HOME` from `Config.Env`.
  No change needed. Add a test asserting the projected JSON never contains a secret value
  even when `Config.Env` does (regression guard against a future field addition).
- **(iv) S4 — secret VALUES MUST NEVER enter any hash or the state file.** The provision
  step hash (`provision.rs:38-45`) is `sha256("shell\n" + normalize(run))` — it hashes the
  user-authored `run` string, which does NOT contain secret values (secrets arrive via
  env, not text substitution). The fingerprint (§7) hashes only `{KEY, persist, from}`
  metadata, never values. `ProvisionState` (`state_store.rs:15-24`) stores step hashes and
  package names — add the secrets/env fingerprint (metadata only). Assert (test): the
  serialized `provision.json` never contains any secret value.

## 7. Diff / idempotency (value-free)

### 7.1 The fingerprint

Define a **value-free secrets/env fingerprint**: the ordered (BTreeMap key order) set of
records `{ key, persist, from }` for `[secrets]` plus `{ key }` for `[env]` (env values
are intentionally EXCLUDED from the fingerprint so changing `EDITOR = "vim"` →
`"nvim"` re-converges via the env injection, not via a recreate — env is mutable in the
exec/enter path; see precedence note). Concretely:

```
env_secret_fingerprint = sha256(
  "secrets\n" + for each (k,e) in sorted(secrets): "{k}\t{e.persist}\t{e.from}\n"
  + "env\n"   + for each k in sorted(env.keys()):   "{k}\n"
)
```

No values appear (S4 / D0 honored). Store it in `ProvisionState` as a new field
`env_secret_fingerprint: String` (`state_store.rs` ProvisionState, `#[serde(default)]` for
back-compat with existing `provision.json` files that lack it).

### 7.2 It MUST participate in convergence (do NOT repeat the box.home blind spot)

This is the explicit anti-pattern guard: the v2.0 sandbox-diff is "assumed-unchanged"
(`diff.rs:158-161`) and box.home WAS a silent never-converging field (now fixed with a
Warn class). **The fingerprint must be ACTUALLY CHECKED, not assumed-unchanged.**

Wiring in `core::apply` (`core/mod.rs:826`), after the live diff and BEFORE deciding
skip-vs-run:

1. Compute `current_fp = env_secret_fingerprint(&bf)`.
2. Read `stored_fp` from `ProvisionState` (the guest `provision.json`, via the existing
   `store.read`).
3. **Classify the fingerprint delta:**
   - `current_fp == stored_fp` → no env/secret change; proceed normally.
   - `current_fp != stored_fp`:
     - If the delta involves ONLY persist=false secrets or `[env]` keys (added/removed/
       toggled) → **Incremental**: it re-converges by re-running provision with the new
       provision-env (persist=false secrets are injected per-enter, env is per-enter), and
       by the next create/enter picking up env. No recreate needed. After a successful
       apply, write `current_fp` into state.
     - If the delta involves ANY persist=true secret — i.e. a persist=true KEY was ADDED,
       REMOVED, or a secret FLIPPED `persist` true→false or false→true → **force
       Recreate** (a `DiffField { field: "secrets", class: "Recreate" }`). Rationale:
       `Config.Env` is FIXED at `distrobox create` time (same immutability class as image
       /mounts/home — confirmed by the v2.0 grounding that container env/labels are
       immutable post-create). Adding/removing a persisted value or flipping persistence
       changes the required `Config.Env`, which only a destroy+recreate can apply.
   - The recreate classification feeds the EXISTING `DiffResult.class == "Recreate"`
     machinery (`core/mod.rs:850`), so `apply` without `--recreate` refuses with exit 65
     and the cozy "needs a recreate" message already in place (`core/mod.rs:852-869`),
     listing `secrets` as the forcing field. With `--recreate`, the existing destroy+
     recreate flow (`core/mod.rs:872-899`) re-runs create with the new persisted set.

4. **To compute the delta class** the diff needs the PRIOR `[secrets]` shape, not just the
   prior hash. Store, alongside `env_secret_fingerprint`, a small
   `secret_specs: Vec<{key, persist, from}>` and `env_keys: Vec<String>` in
   `ProvisionState` (metadata only, no values). The diff compares prior `secret_specs`
   vs current to decide whether any persist=true key was added/removed/flipped (Recreate)
   vs only persist=false/env changed (Incremental). This makes the check EXPLICIT and
   auditable — the opposite of assumed-unchanged.

> Self-check (Test-phase, structural): the fingerprint is checked in `apply`'s main path
> AND the recreate path resets it (a fresh box gets `current_fp` written after the first
> successful provision). A new box created by `up`/`create` with secrets has no prior
> state; `apply_fresh`/`create` writes the fingerprint after provisioning so the FIRST
> subsequent `apply` is a no-op (no spurious re-converge). This must be tested (§9
> AC-DIFF-3) to prove it is not a write-once-never-read field like the original
> `boxfile_sha` (which, per grounding, is declared but never written — DO NOT replicate
> that).

### 7.3 Value rotation is OUT of diff scope (explicit user workflow)

Rotating a secret's VALUE (same KEY, same persist) does NOT change the fingerprint (by
design — values are never in the fingerprint, D0/S4). So `cbox apply` will report "no
change" after a value rotation. The user-visible workflow, documented in README and the
`secret set` success hint:

- **persist=false** secret rotated: run `cbox secret set <box> <KEY>` then simply
  re-enter / re-run the provision step (`cbox apply <box> --redo <idx>` or
  `cbox enter`) — the new value is read from the keyring at the next exec session.
- **persist=true** secret rotated: run `cbox secret set <box> <KEY>` then
  `cbox apply <box> --recreate` (the persisted value is fixed in `Config.Env` at create,
  so a recreate is required to re-bake it).

`cbox secret set` SHOULD print this hint after storing a value (human mode), and the
README MUST document it as the canonical rotation path. This is a deliberate scope
boundary, called out so it is not mistaken for a bug.

## 8. Doctor

**DECISION: yes — add a non-fatal Secret Service availability line to `cbox doctor`.**

- Extend `DoctorResult` (`spec.rs:190`) with `keyring: KeyringStatus { available: bool,
  detail: String }`.
- In `core::doctor` (`core/mod.rs:567`), probe by attempting a cheap, non-destructive
  keyring round-trip via `SecretStore`: `get("__cbox_doctor__", "__probe__")` — `Ok(_)`
  (Some or None) means the Service is reachable; `Err(Unavailable)` means it is not. This
  performs NO write and stores NO secret.
- It is **non-fatal**: an unavailable keyring does NOT make doctor return Err or set
  `ok=false` (doctor's `ok` stays gated on distrobox+backend per `core/mod.rs:576`). The
  keyring line is informational: `keyring: available=no — Secret Service unreachable;
  secrets will refuse until you unlock it.` This guides the user before they hit the
  exit-75 refusal at create time.
- `render_doctor` (`output.rs:170`) prints the line; `--json` includes the `keyring`
  object. doctor takes the `SecretStore` as a new param (real impl in `main.rs`, mock in
  tests).

## 11. Residual risk & open questions (human ruling before / during implementation)

- **OQ-1 (crate vendoring):** Confirm `keyring` v3 + `secret-service` + `zbus` resolve and
  build inside the offline cbox-dev container image (`make build`). If the offline build
  cannot fetch them, the maintainer must pre-populate the cargo cache volume or vendor.
  *Ruling needed only if `make build` fails.*
- **OQ-2 (`list` enumeration):** `keyring` v3's collection-search API for `list(box)` may
  not expose a clean enumeration under the chosen features. Fallback specified in §3.4
  (intersect Boxfile keys with keyring probes). *Ruling needed only if the clean API is
  unavailable.*
- **OQ-3 (lean / TUI-off build):** `make lint-lean` builds with `--no-default-features`.
  The keyring/secret code must compile in the lean config too (it is not behind the `tui`
  feature). Ensure `rpassword`/`keyring` are unconditional deps, not feature-gated. Low
  risk, just a build-matrix note.
- **RR-1 (host root still sees persisted secrets):** `persist=true` values are in
  `Config.Env`; `podman inspect` as the host user shows them. This is INHERENT to D2
  availability and documented (§6.ii). Not a defect — but the README MUST state it so users
  choose `persist=false` for the strictest secrets. No code mitigation possible without
  abandoning the persisted-availability guarantee.
- **RR-2 (process env visibility):** while cbox runs, resolved secret values are in the
  cbox process's memory and on the child's env; `/proc/<pid>/environ` of the distrobox
  child is readable by the same user. This is the standard, unavoidable cost of env-based
  injection and is strictly better than argv (world-visible in `ps`). Acceptable; note in
  README.
- **RR-3 (`from = "prompt"` deferral):** V3 rejects it now. When prompt-as-source ships
  later, the fingerprint and resolution paths must learn it — flagged so the deferral is
  intentional, not forgotten.
