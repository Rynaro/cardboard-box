# v4.0 — Release Pipeline Specification

**Status:** decision-ready · **Author:** SPECTRA · **Date:** 2026-06-17
**Consumer:** Vivi (implementation) · **Reviewer:** maintainer (Henlavezzo)
**Target repo:** github.com/Rynaro/cardboard-box · default branch `main`

---

## 0. Summary

Add an automated release pipeline to `cbox` so that landing Conventional Commits on `main`
produces (1) an automatically-maintained Release PR carrying the next SemVer version +
`CHANGELOG.md`, and (2) on merge of that PR — a Git tag, a GitHub Release, and **four**
prebuilt Linux tarballs plus a `SHA256SUMS` file, all attached to the Release.

The whole thing is **one workflow file** (`.github/workflows/release.yml`) so the well-known
`GITHUB_TOKEN`-does-not-trigger-downstream-workflows gotcha is sidestepped without a PAT.
`release-please` owns the version; the build jobs are gated on its output and run in the
**same workflow run** that cut the release.

This spec does **not** touch the existing `ci.yml` (the `make check` mirror); CI and release
are orthogonal workflows.

### Locked decisions (carried in, not re-litigated)

| ID | Decision |
|----|----------|
| **L1** | Versioning via `googleapis/release-please-action`, `release-type: rust` (bumps `Cargo.toml` + `Cargo.lock`, generates `CHANGELOG.md`). SemVer driven by Conventional Commits. |
| **L2** | Artifacts: musl **and** glibc, for x86_64 **and** aarch64 → 4 tarballs `cbox-<version>-<target>.tar.gz` + `SHA256SUMS`. Release binary built with `tui` feature on (default). Each tarball: `cbox` + `README.md` + `LICENSE`. |
| **L3** | Linux only. No macOS/Windows targets. |

### Decisions resolved by this spec (the hard parts — see §3)

| ID | Question | Resolution (default) |
|----|----------|----------------------|
| **D1** | `GITHUB_TOKEN` downstream-trigger gotcha | **Single workflow**, build jobs gated on `release-please` `outputs.releases_created` via `needs:` + `if:`. No PAT. |
| **D2** | Cross-compilation method for 4 targets | **`cargo-zigbuild` on a single `ubuntu-latest` x86_64 runner** for all 4 targets. Native-ARM-runner alternative documented as fallback. |
| **D3** | Version bootstrap from `0.1.0` with non-conventional history | Seed `.release-please-manifest.json` at **`0.1.0`** + `release-please-config.json`. First Release PR computes from `0.1.0` forward; the first cut will be **`0.2.0`** (feat) or **`0.1.1`** (fix) depending on the first conventional commit. |
| **D4** | Permissions + repo settings | Workflow `permissions:` blocks specified; maintainer must toggle **"Allow GitHub Actions to create and approve pull requests"**. Full checklist in §3.4. |

## 1. Scope

### In scope
- One new workflow: `.github/workflows/release.yml` (release-please + matrix build + asset upload).
- `release-please` bootstrap files: `release-please-config.json`, `.release-please-manifest.json`.
- `RELEASING.md` documenting the Conventional Commits → SemVer mapping, the Release PR flow, manual/emergency release, and artifact verification.
- A `LICENSE` file (MIT) — currently missing; required by L2 (tarball contents) and already referenced by `Cargo.toml` (`license = "MIT"`) and `README.md`.
- A short CONTRIBUTING note (or section appended to `README.md`/`RELEASING.md`) on the commit convention.
- Acceptance criteria (§5) and validation gates / risks (§6).

### Out of scope
- Publishing to crates.io (not requested; `cbox` is a binary, not a library consumers depend on). Deferred.
- Packaging for distros (AUR, Homebrew tap, `.deb`/`.rpm`, Nix). Deferred.
- Code signing / cosign / SLSA provenance attestation. Deferred (noted as a future hardening in §6).
- Container image publishing (the `cbox-dev` toolchain image is dev-only). Out.
- Changing `ci.yml` or the `make check` gate set.

### Deferred (explicitly, for a later version)
- D-DEF-1: crates.io publish job (gate on a `publish` flag in config).
- D-DEF-2: `cargo-dist` migration if matrix maintenance becomes painful.
- D-DEF-3: SBOM + signing.

## 3. Resolved design decisions (the hard parts)

### 3.1 D1 — The `GITHUB_TOKEN` downstream-trigger gotcha

**Problem.** A GitHub Release (or tag) created by a job authenticated with the default
`secrets.GITHUB_TOKEN` does **not** emit events that trigger other workflows. So a naive
two-file design — `release-please.yml` cuts the release, then a separate
`on: release: { types: [published] }` workflow builds binaries — **silently never fires**.
This is intentional GitHub behaviour (loop-prevention) and the single most common cause of
"my release has no assets."

**Decision: single workflow, output-gated jobs. No PAT.**

One file, `.github/workflows/release.yml`, with three logical stages in the **same run**:

1. `release-please` job — runs the action, exposes its outputs.
2. `build` job (matrix ×4) — `needs: release-please`, gated `if: needs.release-please.outputs.release_created == 'true'`. Cross-builds the 4 tarballs + per-target checksum.
3. `upload` (folded into `build`, or a final `publish` job) — uploads assets to the just-created Release by tag, then assembles + uploads the combined `SHA256SUMS`.

Because all three are in one run that was itself triggered by `push` to `main` (an
event the `GITHUB_TOKEN` *is* allowed to start), there is **no second workflow to trigger** —
the gotcha cannot bite.

**Exact outputs consumed.** `googleapis/release-please-action@v4` exposes (relevant subset):

| Output | Type | Meaning |
|--------|------|---------|
| `release_created` | `'true'`/`''` | A GitHub Release was created on this run (i.e. the Release PR was merged). |
| `releases_created` | `'true'`/`''` | Manifest-mode plural alias; same truthiness for our single-package repo. |
| `tag_name` | string | e.g. `v0.2.0` — the tag to upload assets against. |
| `version` | string | e.g. `0.2.0` (no `v`). Used in tarball names `cbox-0.2.0-<target>.tar.gz`. |
| `upload_url` | string | Release upload URL (optional; `gh release upload` by tag is simpler). |

> **Note on output names:** v4 of the action (manifest mode) is the target. It emits both
> `release_created` (top-level boolean) and per-path/plural variants. Vivi: gate on
> `release_created`; if the action version in use only emits `releases_created`, switch the
> `if:` accordingly. Both are specified so there is no guessing — pin the action to a
> known major (`@v4`) and confirm the output name in that version's README at implement time.
> This is the one place to **verify against the pinned action version**, not invent.

**`needs:` / `if:` wiring (normative shape — see §4 YAML for the literal):**

```yaml
jobs:
  release-please:
    outputs:
      release_created: ${{ steps.rp.outputs.release_created }}
      tag_name:        ${{ steps.rp.outputs.tag_name }}
      version:         ${{ steps.rp.outputs.version }}
  build:
    needs: release-please
    if: ${{ needs.release-please.outputs.release_created == 'true' }}
    strategy: { matrix: { target: [<4 targets>] } }
  publish-checksums:
    needs: [release-please, build]
    if: ${{ needs.release-please.outputs.release_created == 'true' }}
```

**Rejected alternative — PAT (Personal Access Token / fine-grained / GitHub App token).**
You *can* have release-please create the release with a PAT so the `release: published`
event fires a separate build workflow. Rejected because:
- Adds a secret to manage, rotate, and scope (`contents:write`, `pull-requests:write`); a leaked PAT is worse than a leaked `GITHUB_TOKEN` (longer-lived, broader).
- Two-workflow indirection is harder to reason about and to dry-run.
- Buys nothing we need: the single-workflow pattern already builds in the same run.
- A GitHub App token is the "correct" heavyweight answer but is overkill for a single-maintainer OSS repo.

The single-workflow pattern is the canonical, secret-free solution and is what `release-please`'s own docs recommend for exactly this case.

### 3.2 D2 — Cross-compilation method for the 4 targets

**Targets (L2):**
`x86_64-unknown-linux-gnu`, `x86_64-unknown-linux-musl`,
`aarch64-unknown-linux-gnu`, `aarch64-unknown-linux-musl`.

**Decision: `cargo-zigbuild` on a single `ubuntu-latest` (x86_64) runner, matrix over the 4 targets.**

`cargo-zigbuild` uses `zig cc` as the cross-linker. It handles all four targets from one
x86_64 host: musl static linking and — crucially — **glibc version pinning** for the gnu
targets (so the binary runs on older glibc than the runner's). Because `cbox` has **zero
build-time native dependencies** (it only *shells out* to distrobox/podman/docker at
runtime; ratatui + crossterm are pure Rust), there is no C toolchain / sysroot pain — zig
supplies everything needed. This is the lowest-moving-parts option that satisfies L2 + L3.

**Why this over the alternatives:**

| Approach | Verdict | Reasoning |
|----------|---------|-----------|
| **(b) `cargo-zigbuild`, 1 runner** | **SELECTED** | One runner type (`ubuntu-latest`), one tool, all 4 targets. glibc pinning prevents "GLIBC_2.xx not found" on user machines. Pure-Rust deps mean zig has nothing to fight. Fast, cheap, deterministic. |
| (a) Native runners (`ubuntu-latest` + `ubuntu-24.04-arm`) | Fallback | Most "obviously correct" (native gnu + native musl via `musl-tools`). BUT depends on `ubuntu-24.04-arm` runner availability — **free for public repos** per GitHub's Jan-2025 GA announcement, but availability/quotas can change; if unavailable, aarch64 stalls. Also pins glibc to the runner's (newer) glibc unless extra care is taken. Documented as the fallback if zigbuild ever misbehaves. |
| (c) `cross` | Rejected | Spins up per-target Docker containers (Docker-in-Actions); slower, heavier, and overkill when there are no native build deps. zigbuild is strictly simpler for this dependency profile. |

**Toolchain setup (normative):**

1. `actions/checkout@v4`.
2. Install Rust + the 4 targets:
   `rustup target add x86_64-unknown-linux-gnu x86_64-unknown-linux-musl aarch64-unknown-linux-gnu aarch64-unknown-linux-musl` (the matrix adds only the per-job target; listed here for completeness).
3. Install zig: `mlugg/setup-zig@v1` (or `goto-bus-stop/setup-zig`) — pins a zig version.
4. Install cargo-zigbuild: `cargo install cargo-zigbuild` (or `taiki-e/install-action@cargo-zigbuild` for a prebuilt, faster).
5. Build: `cargo zigbuild --release --target <matrix.target>` — **no `--features` flag** because `tui` is default-on (L2 says build with tui on; the default already does that). Do **not** pass `--all-features` (that would also enable `testkit`, which is test-only).
6. Strip: `cargo-zigbuild` does not auto-strip; either add `strip = true` to a `[profile.release]` in `Cargo.toml` (cleanest, reproducible, applies everywhere) **or** strip in CI. For aarch64 the host `strip` won't work on ARM binaries, so prefer the Cargo profile setting — see DECISION below.
7. Output binary path: `target/<target>/release/cbox`.

**`tui` feature / musl confirmation.** `ratatui` 0.28 and `crossterm` 0.28 are pure Rust
(crossterm uses `rustix`/`libc` syscall bindings, not a C library that needs linking). No
`*-sys` crate in the dependency tree links a system C library at build time. Therefore
`x86_64-unknown-linux-musl` and `aarch64-unknown-linux-musl` produce **fully static** binaries
with no runtime shared-library dependency. **Assumption A-1:** the lockfile contains no
hidden `*-sys` crate pulling a C dep. Vivi MUST verify with `cargo tree -e features` /
`cargo tree | grep -i sys` during implementation; this is cheap and catches the one risk
that would break musl (see R-1, §6).

**Stripping — DECISION (D2-strip):** set in `Cargo.toml`:
```toml
[profile.release]
strip = true        # strip symbols from the release binary (smaller tarballs)
```
Rationale: target-agnostic (works for ARM cross-builds where host `strip` can't read the
binary), reproducible, and visible in source rather than hidden in CI. Optionally also add
`lto = "thin"` and `opt-level = "z"`/`"s"` for size — **recommended default: `strip = true`
only**, leave `lto`/`opt-level` as-is to avoid changing runtime behaviour/perf in a release-
infra PR (keep the PR's blast radius tight). Flagged as OQ-2 if the maintainer wants size LTO.

**Tarball assembly (per matrix target):**
```
stage cbox  README.md  LICENSE  →  cbox-<version>-<target>.tar.gz
sha256sum cbox-<version>-<target>.tar.gz > cbox-<version>-<target>.tar.gz.sha256
```
Then a final job concatenates the 4 per-target `.sha256` lines into one `SHA256SUMS`
(format: `<hash>␠␠<filename>`, exactly what `sha256sum -c` expects) and uploads it + the 4
tarballs to the Release.

### 3.3 D3 — Version bootstrap

**Situation.** `Cargo.toml` = `0.1.0`, **no** prior GitHub releases, **no** release-please
state files, and the existing commits are **not** Conventional Commits. release-please needs a
known starting point; otherwise on first run it would either (a) try to parse all history and
produce a noisy/garbage changelog, or (b) not know the current version.

**Decision: manifest-mode bootstrap, seed at `0.1.0`.**

Create two files in the repo root:

**`.release-please-manifest.json`** — the seed:
```json
{
  ".": "0.1.0"
}
```
This tells release-please "the current released version is 0.1.0; compute the next version
from commits landed **after** this is committed." Because the manifest declares the baseline,
release-please does **not** walk pre-existing history to invent a changelog — it only
considers commits since the bootstrap. This is precisely how you avoid the non-conventional
history problem.

**`release-please-config.json`** — the config:
```json
{
  "$schema": "https://raw.githubusercontent.com/googleapis/release-please/main/schemas/config.json",
  "release-type": "rust",
  "packages": {
    ".": {
      "release-type": "rust",
      "package-name": "cbox",
      "changelog-path": "CHANGELOG.md",
      "include-component-in-tag": false,
      "bump-minor-pre-major": true,
      "bump-patch-for-minor-pre-major": false
    }
  }
}
```

Notes on the config knobs:
- `release-type: rust` → bumps `Cargo.toml` `version` **and** `Cargo.lock`, generates `CHANGELOG.md` in Keep-a-Changelog style.
- `package-name: cbox` → used in the Release title / changelog header.
- `include-component-in-tag: false` → tags are clean `vX.Y.Z` (not `cbox-vX.Y.Z`), matching a single-package repo.
- `bump-minor-pre-major: true` → while `0.x`, a **breaking change** bumps the **minor** (→ `0.(y+1).0`) rather than major, the conventional pre-1.0 contract. (When the maintainer is ready for 1.0, they cut it deliberately — see RELEASING.md "going 1.0".)
- `bump-patch-for-minor-pre-major: false` → a `feat:` in `0.x` bumps **minor** (`0.1.0`→`0.2.0`), a `fix:` bumps **patch** (`0.1.0`→`0.1.1`). This is the default and the intuitive behaviour. (Set `true` only if you want feats to be patch-level pre-1.0 — not recommended; we want feats visible as minors.)

**`bootstrap-sha` — not needed.** `bootstrap-sha` is for telling release-please to ignore
history before a given commit when you are *not* seeding the manifest. Since we seed the
manifest at `0.1.0`, the manifest baseline already scopes history. **Decision:** rely on the
manifest seed; do **not** set `bootstrap-sha`. (If, on first run, release-please still picks up
unwanted old commits, the fallback is to add `"bootstrap-sha": "<sha of the bootstrap commit>"`
to the config — documented as a contingency, not the default path.)

**What the FIRST release version will be.**
- The bootstrap commit itself (adding these files + the workflow) should be a `chore:` (or `ci:`/`build:`) — a **non-version-bumping** type — so it does **not** by itself trigger a release. After it merges, release-please opens (or updates) a Release PR only once a version-bumping commit lands.
- The **first cut** depends on the first Conventional Commit that lands after bootstrap:
  - first `feat: …` → Release PR proposes **`0.2.0`** (minor bump from `0.1.0`).
  - first `fix: …` (and no feat) → Release PR proposes **`0.1.1`** (patch bump).
  - a breaking change (`feat!:` / `BREAKING CHANGE:`) pre-1.0 → still **`0.2.0`** (minor, per `bump-minor-pre-major`).
- **Recommended default for the very first release:** make the bootstrap PR itself the
  release-infra change as `ci:`/`chore:` (no bump), then the maintainer's next real change
  drives `0.2.0`/`0.1.1` naturally. Vivi does **not** need to hand-pick a first version —
  the seed makes it deterministic.

> **Explicit instruction to Vivi:** seed the manifest at `0.1.0`. Do **not** set the
> `Cargo.toml` version manually after this — release-please owns it from now on. Do **not**
> guess a first version; let the first post-bootstrap conventional commit compute it.

### 3.4 D4 — Required permissions & repo settings (maintainer checklist)

**Workflow-level `permissions:`.** Set least-privilege at job granularity:

| Job | Permissions | Why |
|-----|-------------|-----|
| `release-please` | `contents: write`, `pull-requests: write` | Create/maintain the Release PR (needs PR write), create the tag + Release on merge (needs contents write). |
| `build` | `contents: read` | Just checks out + builds. (Default-deny everything else.) |
| `publish-checksums` / upload step | `contents: write` | `gh release upload` attaches assets to the Release. |

Set a restrictive top-level `permissions: { contents: read }` and elevate per-job. (See §4
YAML.)

**Repo settings the maintainer MUST toggle (one-time, in GitHub UI):**

- [ ] **Settings → Actions → General → Workflow permissions → enable "Allow GitHub Actions to create and approve pull requests."** Without this, release-please **cannot open the Release PR** (`GITHUB_TOKEN` is blocked from creating PRs by default). This is the #1 first-run failure.
- [ ] Confirm **"Read and write permissions"** is selected for `GITHUB_TOKEN` under the same Workflow permissions panel (or rely solely on the per-workflow `permissions:` blocks — both work; the per-workflow block is preferred and is what this spec ships).
- [ ] Ensure **branch protection on `main`** (if any) does not block the release-please bot from merging / pushing the changelog+version commit. If "require PR reviews" is on, the maintainer reviews+merges the Release PR manually (which is the intended human gate anyway).
- [ ] (If using native-ARM fallback only) confirm `ubuntu-24.04-arm` runners are available to the repo (free for public repos). N/A for the zigbuild default.

No PAT, no GitHub App, no org secrets required for the default (zigbuild + single-workflow) design.

## 4. The workflow (normative reference shape)

> This is the **reference shape** Vivi implements — exact action versions/output names pinned
> at implementation against the chosen action major. The YAML below is illustrative-normative:
> the **structure, gating, permissions, and matrix are binding**; cosmetic step names are not.

```yaml
name: Release

on:
  push:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false   # do NOT cancel a release mid-flight

jobs:
  release-please:
    name: release-please
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    outputs:
      release_created: ${{ steps.rp.outputs.release_created }}
      tag_name: ${{ steps.rp.outputs.tag_name }}
      version: ${{ steps.rp.outputs.version }}
    steps:
      - uses: googleapis/release-please-action@v4
        id: rp
        with:
          # config + manifest files at repo root (default paths)
          # token defaults to GITHUB_TOKEN — fine, single-workflow pattern
          config-file: release-please-config.json
          manifest-file: .release-please-manifest.json

  build:
    name: build ${{ matrix.target }}
    needs: release-please
    if: ${{ needs.release-please.outputs.release_created == 'true' }}
    runs-on: ubuntu-latest
    permissions:
      contents: read
    strategy:
      fail-fast: false
      matrix:
        target:
          - x86_64-unknown-linux-gnu
          - x86_64-unknown-linux-musl
          - aarch64-unknown-linux-gnu
          - aarch64-unknown-linux-musl
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ needs.release-please.outputs.tag_name }}   # build the exact tagged tree
      - name: Install Rust + target
        run: |
          rustup toolchain install stable --profile minimal
          rustup target add ${{ matrix.target }}
      - name: Install zig + cargo-zigbuild
        uses: mlugg/setup-zig@v1
      - run: cargo install cargo-zigbuild --locked   # or taiki-e/install-action
      - name: Build (tui on by default — no --features)
        run: cargo zigbuild --release --target ${{ matrix.target }}
      - name: Stage tarball
        run: |
          V="${{ needs.release-please.outputs.version }}"
          T="${{ matrix.target }}"
          BIN="target/${T}/release/cbox"
          DIR="cbox-${V}-${T}"
          mkdir "$DIR"
          cp "$BIN" README.md LICENSE "$DIR"/
          tar -czf "${DIR}.tar.gz" "$DIR"
          sha256sum "${DIR}.tar.gz" > "${DIR}.tar.gz.sha256"
      - uses: actions/upload-artifact@v4
        with:
          name: dist-${{ matrix.target }}
          path: |
            cbox-*.tar.gz
            cbox-*.tar.gz.sha256

  publish:
    name: publish assets + SHA256SUMS
    needs: [release-please, build]
    if: ${{ needs.release-please.outputs.release_created == 'true' }}
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/download-artifact@v4
        with: { pattern: dist-*, merge-multiple: true, path: dist }
      - name: Assemble SHA256SUMS
        run: cat dist/*.tar.gz.sha256 > dist/SHA256SUMS && sed -i 's# dist/# #' dist/SHA256SUMS
      - name: Upload to release
        env: { GH_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
        run: |
          gh release upload "${{ needs.release-please.outputs.tag_name }}" \
            dist/*.tar.gz dist/SHA256SUMS --clobber
```

Implementation notes (binding):
- `concurrency.cancel-in-progress: false` — never cancel a release that is uploading assets.
- `--clobber` on `gh release upload` makes re-runs **idempotent** (re-uploading overwrites rather than failing on "asset already exists"). Addresses R-4 (§6).
- Per-target `.sha256` files use bare filenames (no `dist/` path prefix) in `SHA256SUMS` so `sha256sum -c SHA256SUMS` works when run from the directory holding the tarballs.
- `checkout` the **tag** (`needs.release-please.outputs.tag_name`) in build jobs so the binary embeds the released version, not a later main commit.

## 5. Acceptance criteria (GIVEN / WHEN / THEN)

Each AC is tagged with how it is checkable: **[static]** = `actionlint` / YAML lint / dry-run / file inspection (no real tag needed); **[live]** = requires an actual commit/merge cutting a real tag.

| ID | GIVEN | WHEN | THEN | Check |
|----|-------|------|------|-------|
| **AC-1** | Bootstrap merged; a `feat: …` commit lands on `main` | release-please runs | A Release PR exists proposing version **`0.2.0`** (minor) with a `CHANGELOG.md` entry and updated `Cargo.toml`/`Cargo.lock` | [live] |
| **AC-2** | A `fix: …` commit lands (no feat) | release-please runs | Release PR proposes **`0.1.1`** (patch) | [live] |
| **AC-3** | A `feat!:` / `BREAKING CHANGE:` commit lands (pre-1.0) | release-please runs | Release PR proposes a **minor** bump (`0.x`→`0.(x+1).0`), not major (per `bump-minor-pre-major`) | [live] |
| **AC-4** | An open Release PR | the PR is **merged** | A tag `vX.Y.Z` and a GitHub Release are created in the **same** workflow run | [live] |
| **AC-5** | A Release was just created | the `build` matrix runs | Exactly **4** tarballs are produced: `cbox-<v>-{x86_64,aarch64}-unknown-linux-{gnu,musl}.tar.gz` | [live] |
| **AC-6** | The 4 tarballs exist | `publish` runs | A `SHA256SUMS` file is attached to the Release alongside the 4 tarballs (5 assets total) | [live] |
