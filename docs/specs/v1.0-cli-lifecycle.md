## 3. COMMAND SURFACE (Phase 1 ŌĆö normative)

### 3.0 Global flags & conventions

| Flag | Type | Default | Meaning |
|---|---|---|---|
| `--json` | bool | false | Machine output (NDJSON-free single JSON document per invocation). Honored where a stable schema exists: `list`, `inspect`, `create` (result), `doctor`. |
| `-q, --quiet` | bool | false | Suppress cozy chrome; errors still print to stderr. |
| `-v, --verbose` | count | 0 | `-v` = show the exact spawned argv; `-vv` = stream child stdout/stderr live. |
| `--no-color` | bool | auto | Disable ANSI. Auto-disabled when stdout is not a TTY or `NO_COLOR` is set. |
| `-y, --yes` | bool | false | Assume "yes" to confirmations (maps to distrobox `--yes` where relevant). |
| `--dry-run` | bool | false | Print the `distrobox`/backend argv that *would* run; execute nothing. Maps to distrobox `--dry-run` for `create`. |
| `--backend <podman\|docker>` | enum | auto | Override backend detection (overrides `$CBOX_BACKEND`). |

**Conventions:**
- Human output is the default; `--json` is opt-in and emits exactly one JSON object/array to stdout (logs/chrome go to stderr, so `cbox ŌĆ” --json | jq` is always clean).
- All destructive actions (`rm`) confirm unless `-y`.
- Exit codes are a fixed taxonomy (┬¦3.8). Never `exit(1)` for everything.
- A box NAME is validated client-side against `^[a-zA-Z0-9][a-zA-Z0-9_.-]*$` before any spawn (fail fast, exit 64).

### 3.1 `cbox create`

Create a box, imperatively (flags) or declaratively (from a Boxfile).

```
cbox create [NAME] [flags]
cbox create --file Boxfile.toml          # declarative (Phase 2 wires provisioning; P1 honors name/image/packages/docker/mounts)
```

| Arg/Flag | Type | Default | Maps to |
|---|---|---|---|
| `NAME` (positional) | string | ŌĆö (required unless `--file` supplies it) | `distrobox create --name <NAME>` |
| `-i, --image <IMG>` | string | `registry.fedoraproject.org/fedora-toolbox:latest` | `--image <IMG>` |
| `-p, --package <PKG>` (repeatable) | list | `[]` | `--additional-packages "<pkg pkg ŌĆ”>"` (joined space-sep) |
| `-m, --mount <H:G[:mode]>` (repeatable) | list | `[]` | one `--volume <H:G[:mode]>` per entry |
| `--docker <none\|host\|nested>` | enum | `none` | expands to flags per ┬¦5 (the spectrum) |
| `--home <DIR>` | path | unset | `--home <DIR>` |
| `--hostname <H>` | string | unset | `--hostname <H>` |
| `--init` | bool | false | `--init` (systemd; implies `--unshare-process`) |
| `--pull` | bool | false | `--pull` (implies distrobox `--yes`) |
| `--root` | bool | false | `--root` (pass-through; A6 ŌĆö not a Boxfile field in v1.0) |
| `--file <PATH>` | path | unset | read Boxfile, derive all of the above (Boxfile flags lose to explicit CLI flags) |
| `--dry-run` | bool | false | `distrobox create --dry-run ŌĆ”` ŌåÆ prints backend `create` line, no container made |

**Canonical spawn (imperative, `--docker none`):**
```
distrobox create --yes --name <NAME> --image <IMG> \
  [--additional-packages "<pkgs>"] [--volume <H:G[:mode]> ŌĆ”] \
  [--home <DIR>] [--hostname <H>] [--init]
```
**Human output (success):**
```
Ō£ō Created box "web-dev" (fedora-toolbox:latest, docker: none)
  Enter it with:  cbox enter web-dev
```
**`--json` output:** `{ "ok": true, "action": "create", "name": "web-dev", "image": "...", "docker": "none", "backend": "podman", "argv": ["distrobox","create",...] }`

> **Phase 1 boundary:** `create --file` in P1 maps Boxfile ŌåÆ create flags (name/image/packages/docker/mounts/home/hostname/init). The **provision step list runs in Phase 2** (┬¦8). P1 must *parse and validate* provision steps but may no-op them behind a `--no-provision` default-on guard so the schema never paints P2 into a corner.

### 3.2 `cbox list`

| Arg/Flag | Type | Default | Behavior |
|---|---|---|---|
| `-a, --all` | bool | false (cbox-managed only when Boxfile metadata exists; else all distrobox boxes) | include non-cbox distroboxes |
| `--json` | bool | false | machine output |

- **Human path:** wrap `distrobox list` (already a clean table) but **re-render** through cbox's own table so chrome is consistent and color-controlled. Columns: `NAME ┬Ę STATUS ┬Ę IMAGE ┬Ę DOCKER ┬Ę CBOX?`. The `DOCKER` and `CBOX?` columns come from container labels cbox stamps at create time (┬¦5.5).
- **Machine path (`--json`):** do **not** parse `distrobox list` text (A2). Run:
  ```
  <backend> ps -a --filter label=manager=distrobox --format json
  ```
  and project each entry to a stable cbox schema (┬¦13.list_json_schema). The `cbox.docker_mode` label (┬¦5.5) supplies the docker-mode column; absent label ŌåÆ `"unknown"`.

**`--json` shape:** `{ "ok": true, "boxes": [ { "name","status","image","docker_mode","cbox_managed","id" } ] }`

### 3.3 `cbox rm` (alias `cbox destroy`)

```
cbox rm <NAME>... [flags]
```

| Flag | Type | Default | Maps to |
|---|---|---|---|
| `-f, --force` | bool | false | `distrobox rm --force` (also stops a running box) |
| `--rm-home` | bool | false | `distrobox rm --rm-home` (remove custom home dir) |
| `-y, --yes` | bool | false | skip the confirmation prompt |
| `--all` | bool | false | `distrobox rm --all` (refuses without `--yes`) |

**Spawn:** `distrobox rm [--force] [--rm-home] <NAME> [<NAME>ŌĆ”]`
**Confirmation (no `-y`):** `Remove box "web-dev"? This deletes the container. [y/N]`
**Human output:** `Ō£ō Removed box "web-dev"`
**`--json`:** `{ "ok": true, "action": "rm", "removed": ["web-dev"], "skipped": [] }`

### 3.4 `cbox enter` (alias `cbox use`)

```
cbox enter <NAME> [-- <CMD>...]
```

| Flag | Type | Default | Maps to |
|---|---|---|---|
| `--root` | bool | false | `distrobox enter --root` |
| `--clean-path` | bool | false | `distrobox enter --clean-path` |
| `-- <CMD>...` | passthrough | login shell | `distrobox enter --name <NAME> -- <CMD>` |

**Spawn:** `distrobox enter --name <NAME> [--root] [--clean-path] [-- <CMD>ŌĆ”]`
**Critical wrapper behavior:** `enter` is **interactive** ŌĆö the wrapper must `exec`/inherit stdio (TTY pass-through), NOT capture stdout/stderr. This is the one command whose runner path bypasses capture (┬¦4.4, `RunMode::Interactive`). `--json` is **rejected** for `enter` (exit 64: "enter is interactive; --json not supported").
**Behavior:** if box does not exist ŌåÆ exit 69 with a cozy hint (`No box named "x". Create it with: cbox create x`). If box is stopped, distrobox auto-starts it (no special handling).

### 3.5 `cbox inspect` (alias `cbox show`)

```
cbox inspect <NAME> [--json]
```

- **Source of truth:** `<backend> inspect <container-id-or-name> --format json` (podman/docker `inspect` both emit JSON). cbox resolves NAME ŌåÆ container via the `manager=distrobox` label set.
- **Human output:** a cozy panel: name, status, image, created, docker-mode (from label), mounts (hostŌåÆguest), packages-at-create (from `cbox.packages` label if present), the backend, and the resolved Boxfile path if one was used.
- **`--json`:** a **stable, projected** cbox schema (NOT raw `podman inspect`, which is huge and backend-specific). Raw is available under a `--raw` escape hatch. Projected schema in ┬¦13.inspect_json_schema.

**`--json` shape:** `{ "ok": true, "box": { "name","status","image","created","docker_mode","mounts":[{"host","guest","mode"}],"packages":[...],"backend","id","boxfile_path":null } }`

### 3.6 `cbox edit`

```
cbox edit <NAME>            # edit the box's associated Boxfile
cbox edit --file <PATH>     # edit a Boxfile directly
```

`edit` is a **Boxfile-centric** operation (not "edit a running container" ŌĆö distrobox has no such concept; containers are immutable-ish). Semantics:

1. Resolve the Boxfile for NAME: look up `cbox.boxfile_path` label ŌåÆ fallback to `$XDG_CONFIG_HOME/cbox/boxes/<NAME>/Boxfile.toml`. If none exists, scaffold one from the live container's inspected state (image, mounts, docker-mode) so `edit` always has something to open.
2. Open it in `$VISUAL` || `$EDITOR` || a safe fallback (`vi`/`nano` probe). Single-binary rule (A7): cbox spawns the editor inheriting the TTY; it does not embed one in P1 (the TUI gets an embedded editor in P3).
3. On editor exit, **re-validate** the TOML against the Boxfile schema (┬¦5). On validation error: show the error with line context, offer `[r]e-edit / [k]eep-anyway / [a]bort`.
4. **P1 stops here** (edit = "change the declarative source"). Applying a Boxfile diff to a live box (recreate) is **`cbox apply`, Phase 2** (┬¦8). `edit` prints: `Saved Boxfile for "web-dev". Apply changes with:  cbox apply web-dev   (Phase 2)`.

`--json` is **not supported** for `edit` (interactive).

### 3.7 `cbox doctor` (supporting, not in the original 7 but required by A1/A8)

Preflight environment check. No box mutation.

- Detects `distrobox` presence + version (warns < floor 1.6, A8).
- Detects backend: podman and/or docker; prints which cbox will use and why (┬¦4.5).
- Checks the user can talk to the backend (`<backend> info` exit 0).
- `--json`: `{ "ok": bool, "distrobox": {"present","version","supported"}, "backend": {"selected","podman","docker"}, "warnings": [...] }`
- Exit 0 if usable, exit 70 if distrobox missing, exit 75 if no usable backend.

### 3.8 Exit-code taxonomy (sysexits-aligned, normative)

| Code | Name | When |
|---|---|---|
| 0 | OK | success |
| 64 | USAGE | bad CLI usage / invalid NAME / `--json` on interactive cmd |
| 65 | DATAERR | Boxfile schema/validation failure |
| 69 | UNAVAILABLE | named box does not exist |
| 70 | SOFTWARE | distrobox binary missing/unusable |
| 75 | TEMPFAIL | backend (podman/docker) unreachable |
| 74 | IOERR | spawn/IO failure capturing child output |
| 125 | BACKEND_NONZERO | wrapped distrobox/backend exited non-zero (code surfaced in error detail) |

Rule: when the wrapped process exits non-zero, cbox exit code is **125** and the child's exit code + last stderr lines are surfaced in the human/JSON error (┬¦4.3). cbox never silently swallows a child failure.

## 4. DISTROBOX PROCESS-WRAPPER DESIGN

### 4.1 The core seam ŌĆö `DistroboxRunner` trait

The entire testability story rests on one trait. **No command handler shells out directly**; every spawn goes through this seam.

```rust
// src/dbox/runner.rs  (signatures only ŌĆö narrative; normative copy in ┬¦13)
pub struct Invocation {
    pub program: String,        // "distrobox" | "podman" | "docker"
    pub args: Vec<String>,
    pub mode: RunMode,
    pub env: Vec<(String, String)>,
}

pub enum RunMode {
    Capture,      // collect stdout/stderr/exit ŌĆö used by create/list/rm/inspect/doctor
    Interactive,  // inherit TTY, no capture ŌĆö used by enter
    DryRun,       // never spawn; return the would-be argv as stdout
}

pub struct CmdOutput {
    pub status: i32,            // -1 if killed by signal
    pub stdout: String,
    pub stderr: String,
    pub argv: Vec<String>,      // exact argv, for -v and --json provenance
}

pub trait DistroboxRunner: Send + Sync {
    fn run(&self, inv: Invocation) -> Result<CmdOutput, RunnerError>;
    fn run_interactive(&self, inv: Invocation) -> Result<i32, RunnerError>; // TTY pass-through
}
```

- **`RealRunner`** (`src/dbox/real.rs`): uses `std::process::Command`. `Capture` mode pipes stdout/stderr; `Interactive` inherits (`Stdio::inherit()`) and `.status()`-waits so signals/TTY work for `enter`.
- **`MockRunner`** (`src/dbox/mock.rs`, behind `#[cfg(test)]` + a `testkit` feature): a programmable double. Queue of `(matcher, response)`. A matcher matches on `program` + an `args`-subsequence; response is a canned `CmdOutput` or error. Records every `Invocation` for assertions (`mock.calls()`).

### 4.2 Why a trait (not just inject a closure)

- CI has no distrobox (constraint). The trait lets **100% of command-handler logic run in CI** against `MockRunner` golden outputs.
- The `argv`-builder (the part most likely to have bugs ŌĆö flag mapping in ┬¦3/┬¦5) is a **pure function** `build_create_argv(&CreateSpec) -> Vec<String>` tested independently of any process (unit tests, no mock even needed).
- `--dry-run` is implemented once in the runner (return argv), so every command gets a free, real, testable dry-run.

### 4.3 Error surfacing (humane)

Two layers:
1. **`RunnerError`** (`thiserror`) ŌĆö spawn-level: binary-not-found (ŌåÆ exit 70), IO error (ŌåÆ 74), interactive-spawn-failure.
2. **Command-level mapping** ŌĆö a handler inspects `CmdOutput.status`. Non-zero ŌåÆ build a `CboxError` (`thiserror`) carrying: the human headline, the child's exit code, and the **last 5 lines of child stderr** (full stderr in `--verbose`/`--json`). Common distrobox failures get pattern-matched to cozy messages:

| distrobox stderr signal | Cozy message |
|---|---|
| `already exists` | `A box named "x" already exists. Use a different name or remove it: cbox rm x` |
| `image ŌĆ” not found` / pull fail | `Couldn't pull image "img". Check the name, or try: cbox create ŌĆ” --pull` |
| `command not found: distrobox` | `distrobox isn't installed or isn't on PATH. See: cbox doctor` |
| backend socket refused | `Can't reach podman/docker. Is the service running? cbox doctor` |

Unmatched failures fall through to a generic surface that **always** includes the exact argv (so users can reproduce manually) ŌĆö never a bare "error: 1".

### 4.4 Capture vs interactive (the one fork)

`enter`/`use` is interactive ŌåÆ `run_interactive` ŌåÆ `Stdio::inherit()` for stdin/stdout/stderr so the user gets a real shell with a real TTY, job control, and signal handling. Everything else is `Capture`. This fork is the only place stdio handling branches; it is covered by an explicit acceptance criterion (┬¦7, AC-ENTER-3) asserting the runner is called with `RunMode::Interactive`.

### 4.5 Backend detection ŌĆö `Backend::detect()`

```
order:
  1. --backend flag        (explicit override)
  2. $CBOX_BACKEND env     (podman|docker)
  3. distrobox's own choice: read $DBX_CONTAINER_MANAGER if set
  4. probe: podman on PATH and `podman info` exit 0  ŌåÆ podman   (preferred)
  5. else docker on PATH and `docker info` exit 0    ŌåÆ docker
  6. else ŌåÆ exit 75 (no usable backend) with doctor hint
```
podman is **preferred** when both are usable (distrobox's own default on rootless Linux). The chosen backend is used only for the **machine read paths** (`list --json`, `inspect`); creation/removal/enter go through `distrobox`, which selects its own manager. cbox passes `DBX_CONTAINER_MANAGER` through to distrobox to keep them aligned.

## 5. THE DOCKER-ACCESS SPECTRUM (`docker = none | host | nested`)

This is the product differentiator: a single, legible knob across the sandboxŌåöhost-docker
spectrum. Each mode is a **named bundle of concrete flags** appended to `distrobox create`.

### 5.0 Honesty preamble (A4)

distrobox containers run, by design, `--privileged --security-opt label=disable` and share
the host user/home/network/IPC/PID namespaces by **default** (verified via `--dry-run` on the
host). So "fully sandboxed" in cbox means **"no *additional* host-resource coupling, plus the
optional `--unshare-*` hardening cbox can add"** ŌĆö not a security sandbox in the gVisor sense.
The spec states this plainly; the modes differ in *coupling to the container runtime*, which is
exactly the axis users care about for "can I use docker here."

### 5.1 `docker = none` (default ŌĆö maximally decoupled)

**Intent:** the box has no special access to any container runtime. Optionally hardened.

| Boxfile sub-field | Flag added to `distrobox create` |
|---|---|
| (base) | *(nothing extra ŌĆö no socket, no DinD package)* |
| `sandbox.unshare = ["netns","ipc","process","devsys","groups"]` (optional list) | one `--unshare-<x>` per entry |
| `sandbox.unshare = "all"` | `--unshare-all` |

**Verification:** inside the box, `docker ps` / `podman ps` should fail with "cannot connect" (no socket, no client configured). cbox stamps label `cbox.docker_mode=none`.

### 5.2 `docker = host` (host-visible: docker runs *in* the box, usable from host)

**Intent (the headline use case):** run `docker`/`podman` commands inside the box, but the
containers they create are the **host's** containers ŌĆö visible and usable from the host. Achieved
by mounting the host container socket into the box and installing a client.

| Boxfile sub-field | Flag / step |
|---|---|
| socket mount (podman host) | `--volume /run/user/$UID/podman/podman.sock:/run/user/$UID/podman/podman.sock` |
| socket mount (docker host) | `--volume /var/run/docker.sock:/var/run/docker.sock` |
| client package | add `docker-cli` (or `podman-remote`) to `--additional-packages` |
| env hint | `--additional-flags "--env DOCKER_HOST=unix:///var/run/docker.sock"` (docker case) |

cbox picks the socket based on the **detected backend** (┬¦4.5): podman socket for podman hosts,
docker socket for docker hosts. The socket path is resolvable at create time and recorded.
**Verification:** inside the box, `docker ps` lists the **host's** containers; a container started
in the box appears in `docker ps` *on the host*. cbox stamps `cbox.docker_mode=host`.

> **Confirmed on host:** an existing box on this machine already mounts
> `/var/run/docker.sock` and installs `docker-cli` ŌĆö this mode mirrors that real,
> working configuration.

### 5.3 `docker = nested` (Docker-in-Docker, isolated runtime inside the box)

**Intent:** a **private** docker daemon lives inside the box. Containers it creates are *not*
visible to the host ŌĆö full nesting/isolation of the container runtime.

| Boxfile sub-field | Flag / step |
|---|---|
| privileged (already default) | distrobox default `--privileged` suffices; we add `--init` so an in-box dockerd can be managed by systemd |
| package | add `docker-ce` + `docker-ce-cli` + `containerd` (or `podman` for rootless nested) to `--additional-packages` |
| provision (P2) | enable + start `dockerd` via the provision step list; in P1 this is a documented post-create manual step or `--init` + a systemd unit |
| no host socket | **explicitly do not** mount any host socket |

**Verification:** inside the box, `docker ps` talks to the box's *own* daemon; host `docker ps`
does **not** show the box's inner containers. cbox stamps `cbox.docker_mode=nested`.
**Note:** true rootless nested docker is finicky; v1.0 documents `--init` + in-box dockerd as the
supported path and flags rootless-nested as a known sharp edge (┬¦10 R5).

### 5.4 Mode comparison (the spectrum, at a glance)

| | `none` | `host` | `nested` |
|---|---|---|---|
| Host socket mounted | no | **yes** | no |
| Container runtime inside box | none | host's (shared) | private daemon |
| Containers visible from host | n/a | **yes** | no |
| Optional `--unshare-*` hardening | yes | (defeats the purpose) | partial |
| Use case | clean dev env | "I want host docker here" | isolated CI-like runtime |

### 5.5 Labels cbox stamps (the metadata backbone)

Every cbox-created box gets these labels via `--additional-flags "--label k=v"`:

| Label | Value | Used by |
|---|---|---|
| `manager` | `distrobox` (distrobox sets this itself) | `list`/`inspect` filtering |
| `cbox.managed` | `true` | `list` CBOX? column |
| `cbox.docker_mode` | `none\|host\|nested` | `list`/`inspect` DOCKER column |
| `cbox.boxfile_path` | abs path or empty | `edit` resolution |
| `cbox.version` | cbox semver that created it | future migrations |

These labels are the durable source of truth that lets cbox reconstruct intent from a live
container without a sidecar database. **No external state store in v1.0** ŌĆö the container labels
+ the Boxfile-on-disk *are* the state.

### 6.2 Validation rules (P1 enforces; exit 65 on failure)
- `name` matches the NAME regex; non-empty.
- `docker` Ōłł {none,host,nested}; `mounts[].mode` Ōłł {ro,rw}; `sandbox.unshare` entries Ōłł enum set or the literal `"all"`.
- `sandbox.unshare` non-empty AND `docker != "none"` ŌåÆ **warning** (hardening contradicts host/nested coupling), not an error.
- mount host/guest are absolute paths; guest paths unique.
- `provision[].type=="shell"` requires `run`; `=="copy"` requires `src`+`dst`.
- Unknown top-level keys ŌåÆ **warning** (forward-compat: lets P2/v1.1 add fields without breaking P1 parsers). Use `serde` with `#[serde(deny_unknown_fields)]` **off** at the top level, on for `[sandbox]`/`[box]` sub-tables.

## 7. ACCEPTANCE CRITERIA (GIVEN/WHEN/THEN ŌĆö mock-testable)

All criteria are written against `MockRunner`; none require a real distrobox. Each maps to a
Rust integration test under `tests/`.

### create
- **AC-CREATE-1** GIVEN a MockRunner that returns exit 0 WHEN `cbox create web-dev -i fedora-toolbox:latest` THEN the runner is called once with program `distrobox` and args containing `["create","--yes","--name","web-dev","--image","fedora-toolbox:latest"]` AND exit code 0 AND stdout contains `Created box "web-dev"`.
- **AC-CREATE-2** GIVEN `--docker host` and a podman backend WHEN create runs THEN args contain a `--volume` entry ending `podman.sock:.../podman.sock` AND `docker-cli` appears in `--additional-packages` AND the box is labeled `cbox.docker_mode=host`.
- **AC-CREATE-3** GIVEN `--docker nested` WHEN create runs THEN args contain `--init` AND `docker-ce` in packages AND **no** host-socket `--volume`.
- **AC-CREATE-4** GIVEN `--docker none` and `sandbox.unshare="all"` (via `--file`) WHEN create runs THEN args contain `--unshare-all`.
- **AC-CREATE-5** GIVEN a MockRunner returning stderr containing `already exists`, exit 125 WHEN create runs THEN exit code is 125 AND stderr shows the cozy "already exists" message AND the exact argv is printed.
- **AC-CREATE-6** GIVEN `--dry-run` WHEN create runs THEN the runner is invoked in `DryRun` mode, no `Capture` spawn occurs, AND stdout is the would-be argv. (Backed by real `distrobox --dry-run` golden file in a non-CI smoke suite.)
- **AC-CREATE-7** GIVEN an invalid NAME `web dev` (space) WHEN create runs THEN exit 64 AND no runner call occurs (fail-fast client-side).

### list
- **AC-LIST-1** GIVEN a MockRunner whose `podman ps ŌĆ” --format json` returns 2 labeled entries WHEN `cbox list --json` THEN stdout is valid JSON with `boxes` length 2 AND each has `name,status,image,docker_mode,cbox_managed,id`.
- **AC-LIST-2** GIVEN the same WHEN `cbox list` (human) THEN a table with header `NAME STATUS IMAGE DOCKER CBOX?` is printed and `docker_mode` from the label appears in the DOCKER column.
- **AC-LIST-3** GIVEN a box lacking the `cbox.docker_mode` label WHEN `cbox list --json` THEN that box's `docker_mode` is `"unknown"` (no crash).

### rm / destroy
- **AC-RM-1** GIVEN box `web-dev` and `-y` WHEN `cbox rm web-dev` THEN runner called with `distrobox rm` args containing `web-dev` AND output `Removed box "web-dev"`.
- **AC-RM-2** GIVEN no `-y` and a mocked "n" on stdin WHEN `cbox rm web-dev` THEN no runner call occurs AND exit 0 with "aborted".
- **AC-RM-3** GIVEN `--force` WHEN rm runs THEN args contain `--force`.
- **AC-RM-4** GIVEN the alias `cbox destroy web-dev -y` THEN behavior is identical to `cbox rm`.

### enter / use
- **AC-ENTER-1** GIVEN box `web-dev` WHEN `cbox enter web-dev` THEN `run_interactive` is called (NOT `run`) with `distrobox enter --name web-dev`.
- **AC-ENTER-2** GIVEN `cbox enter web-dev -- ls -la` THEN args contain `--name web-dev -- ls -la` (passthrough preserved, order intact).
- **AC-ENTER-3** GIVEN `cbox enter web-dev --json` THEN exit 64 with "enter is interactive; --json not supported" AND no runner call.
- **AC-ENTER-4** GIVEN a runner that reports box-not-found WHEN `cbox use missing` THEN exit 69 with the cozy "create it with" hint.
- **AC-USE-1** GIVEN alias `cbox use web-dev` THEN identical to `cbox enter`.

### inspect / show
- **AC-INSPECT-1** GIVEN a MockRunner whose `<backend> inspect ŌĆ” --format json` returns a known container doc WHEN `cbox inspect web-dev --json` THEN stdout is the **projected** schema (name/status/image/created/docker_mode/mounts/packages/backend/id/boxfile_path), NOT raw podman JSON.
- **AC-INSPECT-2** GIVEN `--raw` THEN stdout is the raw backend JSON unmodified.
- **AC-INSPECT-3** GIVEN a non-existent box WHEN inspect runs THEN exit 69 with cozy not-found.
- **AC-SHOW-1** GIVEN alias `cbox show web-dev` THEN identical to inspect.

### edit
- **AC-EDIT-1** GIVEN box `web-dev` with a `cbox.boxfile_path` label pointing at an existing file AND `$EDITOR=true` (no-op) WHEN `cbox edit web-dev` THEN that file is opened (editor spawned with the path) AND on exit the file is re-validated AND "Saved Boxfile" is printed.
- **AC-EDIT-2** GIVEN no existing Boxfile WHEN `cbox edit web-dev` THEN a Boxfile is scaffolded from inspected state (image/mounts/docker_mode present) before the editor opens.
- **AC-EDIT-3** GIVEN the user saves invalid TOML AND chooses `[a]bort` WHEN edit re-validates THEN exit 65 AND the original file is left unchanged.
- **AC-EDIT-4** GIVEN `cbox edit web-dev --json` THEN exit 64 (interactive; json unsupported).

### doctor
- **AC-DOCTOR-1** GIVEN a MockRunner reporting distrobox 1.8.2.4 + podman info exit 0 WHEN `cbox doctor --json` THEN `ok:true`, `backend.selected:"podman"`, `distrobox.supported:true`.
- **AC-DOCTOR-2** GIVEN distrobox absent WHEN `cbox doctor` THEN exit 70 with install guidance.
- **AC-DOCTOR-3** GIVEN distrobox 1.4 (below floor) THEN a warning lists the floor (1.6) but exit is 0 if a backend is usable.

### argv-builder unit layer (no mock needed ŌĆö pure functions)
- **AC-ARGV-1** `build_create_argv(spec)` is a pure function; given a fixed `CreateSpec` it returns a deterministic, asserted `Vec<String>`. One unit test per docker-mode ├Ś sandbox combination. This is the primary defense for the ┬¦5 flag table.

## 8. PHASE 2 SKETCH ŌĆö Boxfile provisioning engine (forward-compat only)

- New command `cbox apply <NAME>` and `cbox up` (create-if-absent + provision).
- The `[[provision]]` list executes **in order** post-create via the existing runner:
  `shell` ŌåÆ `distrobox enter --name <N> -- sh -c "<run>"`; `copy` ŌåÆ hostŌåÆguest file copy.
- **Idempotency:** each provision step gets a content hash; cbox stores applied-hashes as a container label (`cbox.provision.<idx>=<hash>`) so re-`apply` skips unchanged steps ŌĆö reproducible, Vagrant-style.
- `apply` diffs the Boxfile against the live container's labels; image/mount changes require **recreate** (prompt), package/provision changes can be incremental.
- **Seam guarantee from P1:** the Boxfile schema (┬¦6) already carries `provision`, P1 *validates* it, and the runner already supports `enter -- <cmd>` capture ŌĆö so P2 adds an engine, not a new contract.

## 9. PHASE 3 SKETCH ŌĆö TUI dogfooding the CLI (forward-compat only)

- `cbox` (no args) or `cbox tui` launches a `ratatui` app.
- **Hard rule:** the TUI calls the **same internal command functions** the CLI calls ŌĆö it does NOT re-implement flag mapping. CLI handlers are refactored (in P1 already) to a `core::` API: `core::create(spec, &runner) -> Result<CreateOutcome>`, `core::list(&runner)`, etc. The `clap` layer and the TUI are both thin front-ends over `core::`.
- Screens (sketch): box list (table) ŌåÆ detail/inspect panel ŌåÆ create wizard (name/image/packages/docker-mode picker) ŌåÆ confirm-destroy modal ŌåÆ embedded Boxfile editor (the P3 version of `edit`).
- `enter` from the TUI suspends the alt-screen, hands the TTY to `distrobox enter`, restores on exit.
- **Seam guarantee from P1:** because P1 separates `core::` (logic) from the `clap` front-end and routes all spawns through `DistroboxRunner`, the TUI is additive ŌĆö no Phase-1 contract changes.

### 11.2 Layering rule (enforced)

`cli/` may depend on `core/`; `core/` may depend on `dbox/` + `boxfile/`; **nothing** depends on `cli/`. `tui/` (P3) depends on `core/` only ŌĆö never on `cli/`. This is what makes the TUI a pure add-on (┬¦9).

### 11.3 Error-handling strategy
- **Library/core layers** use `thiserror` ŌĆö typed `CboxError` and `RunnerError` enums (each variant carries an exit code). Precise, matchable, no string-typing.
- **`main.rs` boundary** uses `anyhow` only to attach context and render the final message; it maps the typed error's exit code via `std::process::exit(code)`. So: typed inside, ergonomic at the edge.
- One rule: **a wrapped child's non-zero exit is never `?`-bubbled as a Rust error opaquely** ŌĆö it becomes `CboxError::Backend{code, stderr_tail, argv}` so ┬¦4.3 cozy mapping + exit 125 always fire.
