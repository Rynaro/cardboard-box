> **Status:** SHIPPED (v6.0.0) · **Date closed:** 2026-06-21
>
> **Scope:** `cbox export` — imperative command surface wrapping `distrobox-export`; closes the documented gap from v1.0 (README L561, deferred `distrobox-export app integration`).
>
> **Confidence:** Shipped with 20 acceptance tests (AC-EXP-1..16) + 9 argv unit tests passing; `make check` green (fmt + clippy lean/tui + debug + release + all tests).

---

## 0. The gap

v1.0 (spec 2026-06-16, §2.4, L72) deferred "`distrobox-export` app integration" with the rationale that the feature required a deliberate design choice: whether export should be imperative-only or also support a declarative `[[exports]]` Boxfile section participating in the diff/apply engine.

v6.0 ships the **imperative-only** cut. The command surface is `cbox export <BOX> --app|--bin|--service|--list-*` with `--delete` and `--to` modifiers. A declarative seam (the `exports` key in the allowlist) is reserved for future v6.1+ but not wired into the diff/apply engine in v6.0.

---

## 1. Mechanism & design decision

### 1.1 Imperative-only (DECISION-1, recommended)

**Ruling:** Imperative command only; no declarative diff/apply in v6.0.

**Justification:** `distrobox-export` writes host-side files (`.desktop` launchers, wrapper scripts) that cbox **cannot reliably verify from inside the box**. A truthful declarative idempotency engine would need to either (a) accept that `[[exports]]` declarations are write-only hints without host-reality verification, or (b) build a host-side export registry for tracking. Neither fits v6.0's scope.

The imperative command delivers the user value (surface an app NOW) with a fully mock-testable surface, and gives real usage data to inform a declarative design in v6.1.

**Seam:** `src/boxfile/mod.rs` reserves `"exports"` in the `known_top_level` allowlist (one array element). No serde struct, no validation, no apply wiring — only a placeholder signal that future Boxfiles declaring `[[exports]]` will not warn.

**Source:** `.spectra/plans/2026-06-21-v6.0-export.md` §3 DECISION-1; `src/boxfile/mod.rs` line affirms the reservation.

### 1.2 The exec contract

Every export is **imperative** and runs inside the box via distrobox enter:

```
distrobox enter --name <BOX> -- distrobox-export <flags>
```

This shape mirrors `core::enter` and the v1.0 provision-shell pattern. The architecture is:

1. **Precondition:** `core::inspect(box_name)` confirms the box exists (exits 69 if not).
2. **Argv dispatch:** Select the appropriate builder (`build_export_*_argv`) based on the target mode.
3. **Run:** `Invocation::new("distrobox", args, Capture).with_env([backend.dbx_env()])` — same backend-pin as `core::enter` (mod.rs:611).
4. **Failure map:** Distinguish helper-missing (F2, exit 70, cbox-authored message) from other helper errors (F3, exit 125, helper's own stderr).

**Sources:**
- `src/core/mod.rs:577–699` — full `pub fn export` implementation.
- `src/dbox/argv.rs:302–390` — five `build_export_*_argv` builders + unit tests.
- `src/cli/export.rs` — the clap `ExportArgs` + `ArgGroup` enforcement + inter-flag guards.

## 2. Command surface

### 2.1 Synopsis

```bash
cbox export <BOX> --app <APPNAME>                  # surface a GUI app as .desktop
cbox export <BOX> --bin <GUEST_PATH> [--to <DIR>] # surface a binary as wrapper
cbox export <BOX> --service <NAME>                 # surface a systemd service
cbox export <BOX> --app <APPNAME> --delete         # remove previously-exported app
cbox export <BOX> --list-apps                      # list currently-exported apps
cbox export <BOX> --list-bins                      # list currently-exported binaries
```

### 2.2 Clap representation

`ExportArgs` (src/cli/export.rs) uses an `ArgGroup` with `required = true, multiple = false` to enforce exactly one target mode. Values:

- `--app <APPNAME>`: Export a GUI application (creates a host `.desktop` launcher).
- `--bin <GUEST_PATH>`: Export a binary. Requires `--to <HOSTDIR>` (maps to distrobox-export's `--export-path`); if omitted, the helper's own error surfaces (F3, exit 125).
- `--service <NAME>`: Export a systemd service unit.
- `--list-apps`: List apps currently exported from the box (parses stdout).
- `--list-bins`: List binaries currently exported.
- `--delete`: Modifier. Only meaningful with `--app`, `--bin`, `--service`. Combining with list modes → exit 64 (guard in cli/export.rs:75).
- `--to <HOSTDIR>`: Only valid with `--bin`. Omitting it with `--bin` is allowed; distrobox-export will error with its own message (F3).

**Source:** `src/cli/export.rs:14–49`, clap derive macros and in-function guards (lines 68–79).

### 2.3 Exit-code mapping (DECISION-4, §2.4 of the spec)

| Failure | Detection | Exit | Message source |
|---------|-----------|------|---|
| **F1** — Box does not exist | `inspect()` → UNAVAILABLE | 69 | cbox (reuses existing `box_not_found` path) |
| **F2** — `distrobox-export` not found in box | stderr ∋ `distrobox-export` AND (`not found` / `no such file` / `command not found`) | 70 | cbox (actionable: "run `cbox doctor`") |
| **F3** — App/bin not found, missing `--export-path`, or other helper error | exec exit ≠ 0 (catch-all) | 125 | helper's own stderr (shown as tail) |
| **F4** — Backend unreachable / won't start | `RunnerError` or backend error | 125 / via RunnerError | existing backend_error path |
| **F5** — Bad usage (no target, two targets, `--to` w/o `--bin`, `--delete` w/ list) | clap ArgGroup + guards | 64 | clap / cli/export.rs guards |

**Known deviation (OQ-2):** ArgGroup enforcement fires BEFORE cbox's guards, so usage errors from ArgGroup misses (F5: no target / two targets) exit with clap's default usage code **2**, not the spec's intended **64**. The guard-reachable cases (bad box name, `--to` w/o `--bin`, `--delete` w/ list) do return 64. Forcing 64 for all ArgGroup cases would require overriding clap's error handler globally — out of scope for this feature.

**Sources:**
- `src/core/mod.rs:642–661` — F2 and F3 detection logic.
- `src/cli/export.rs:51–79` — inter-flag guards returning exit 64.
- `src/error.rs` — exit-code enum mapping (reuses existing 69, 70, 125 codes).

## 4. Output & rendering

### 4.1 Human (default)

**Mutating success (--app / --bin / --service):**
```
✓ Exported app "firefox" from "dev" to your host menu.
```
or
```
✓ Removed exported app "firefox" for "dev".
```

Pattern: `Exported <mode> "<target>" from "<box>" <verb_suffix>.` where `verb_suffix` is "to your host menu" (app), "as a host wrapper" (bin), or "as a host service" (service).

Optional echo of distrobox-export's own headline in non-quiet mode (provenance, no secrets).

**List success:**
```
firefox
gimp
```
or (empty)
```
No apps exported from "dev".
```

**Rendering code:** `src/cli/export.rs:124–168` (`render_export_human` function).

## 6. Scope boundaries (DECISION rulings)

### D1 — Imperative-only v6.0

**Shipped as:** Imperative command. Declarative `[[exports]]` reserved but not implemented.

**Mitigation:** Seam in `boxfile/mod.rs` allows future additive implementation without breaking v6.0 users.

### D2 — Five-mode MVP surface

**Shipped as:** `--app`, `--bin` (+ `--to`), `--service`, `--list-apps`, `--list-bins`, all with `--delete` modifier where applicable.

**Deferred:** `--export-label`, `--extra-flags`, `--sudo` (available via escape hatch).

### D3 — box→host only

**Shipped as:** Imperative commands surface resources FROM the box TO the host. cbox owns no host-side export registry.

**Corollary:** `--list-apps` / `--list-bins` reflect what `distrobox-export` reports as exported *from that box*. cbox does not enumerate the host filesystem.

### D4 — Exit codes (existing taxonomy reused)

**Shipped:** 64 (usage), 69 (box not found), 70 (helper missing), 125 (helper error). No new codes introduced.

**Known deviation (OQ-2):** Some ArgGroup-level usage errors exit 2 (clap default), not 64. Guard-reachable errors do return 64.

### D5 — Idempotency (N/A for imperative cut)

**Shipped as:** No cbox-side idempotency engine. Users rely on distrobox-export's own behavior (re-export is idempotent; delete-of-absent is benign).

## 7. Residual risks

### RR-1 — List-mode output parsing (version skew)

`distrobox-export --list-apps / --list-binaries` output format may vary across distrobox versions (headers, decoration). v6.0 parses conservatively: split lines, trim, drop empties. Format skew degrades to "show raw output" rather than data loss.

**Mitigation:** The `detail` field in `ExportOutcome` carries the raw helper output for human visibility. Future versions may fingerprint distrobox version at export time if fragility is observed.

### RR-2 — Stopped box auto-start latency

Export on a stopped box incurs distrobox's cold-start cost. No additional timeout added in v6.0 (same as `enter` / provision).

### RR-3 — Orphaned host launchers (future GC)

`cbox rm <BOX>` does not clean host `.desktop` / wrapper files previously exported (cbox owns no host-side registry). Documented honestly in README. Flagged as a possible future `cbox export prune` (host-side garbage collection).

This is not a v6.0 defect—it matches distrobox-export's own lifecycle semantics.

### RR-4 — Deferred flags

`--export-label`, `--extra-flags`, `--sudo` deferred. Escape hatch: `cbox enter <box> -- distrobox-export … [flags]`. v6.1 can add them additively (no rework).

## 8. Assumptions grounded in live testing

| # | Assumption | Risk | Mitigation |
|---|---|---|---|
| A1 | `distrobox-export` is injected at `/usr/bin/distrobox-export` in distrobox-created boxes. | An exec returns "command not found"; we map that to exit 70 with an actionable message. | Confirmed: `src/core/diff.rs:302` already lists `distrobox-export` as an injected binary. |
| A2 | Box must be RUNNING for `distrobox enter -- …` to succeed. | distrobox auto-starts a stopped box on enter; if start fails, backend error surfaces as exit 125. | We do NOT add our own start orchestration; we rely on distrobox's enter-auto-start (same as v1.0's `core::enter`). |
| A3 | `distrobox-export` flags stable across distrobox ≥ 1.6 (documented cbox floor, README:553). | Version skew changes a flag; cbox passes flags verbatim, so skew surfaces as distrobox-export's own error (exit 125). | Verified against real hosts; `cbox doctor` warns on old versions. |
| A4 | Export side effects land on the HOST filesystem. cbox does not read or verify them; success is the exec exit code. | Partial export that exits 0 but wrote nothing — out of cbox's control. | Documented honestly (RR-3). |

**Sources:** Grounded in the shipped implementation (src/core/mod.rs, src/cli/export.rs) and v1.0 precedent.
